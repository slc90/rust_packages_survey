# 深度学习设计方案

## 1. 概述

深度学习模块用于给当前项目提供一组可本地运行的 AI 能力，服务对象主要是：

1. 音频
2. 视频
3. 文本
4. 图像

根据 `docs/requirements.md`，当前目标包含：

1. 本地翻译
2. 人声分离
3. Whisper
4. 图像生成
5. 语音生成

首版目标不是一次性做完整 AI 平台，而是先建立一个统一、清晰、可扩展的本地推理基础层，优先解决：

1. 模型能力的统一接入
2. Bevy UI 可异步调用
3. 输出结果可管理

## 2. 设计结论

建议新增独立 crate：

- `deep_learning`

这个 crate 只负责：

- 模型描述
- 推理任务调度
- 输入输出数据结构
- 模型目录管理
- 统一错误处理
- 不同能力的服务封装

不负责：

- Bevy UI
- 页面状态管理
- 文件弹窗
- 报告导出

这样做的原因很直接：

1. 深度学习推理和 UI 解耦
2. 各能力共享统一运行时，而不是五个功能各做一套
3. 后续更容易替换具体模型实现
4. 方便单元测试和后续独立扩展

## 3. 总体路线

首版建议采用“统一运行时 + 分能力服务”的架构。

建议优先级：

1. `Whisper`
2. `语音生成`
3. `本地翻译`
4. `人声分离`
5. `图像生成`

原因：

1. `Whisper` 和 `语音生成` 更容易快速得到可见结果
2. `本地翻译` 适合作为纯文本任务补齐统一任务流
3. `人声分离` 模型较重，但输入输出边界清楚
4. `图像生成` 工程复杂度和设备要求最高，建议最后接

## 4. 库选型建议

首版建议采用“Rust 主程序 + Rust 原生推理库”的路线。

### 4.1 主推理后端

推荐：

- `candle`

原因：

1. 和当前项目的 Rust 主体更一致
2. 可以尽量避免引入额外运行时桥接层
3. 更适合作为统一推理抽象层的底座
4. 便于把翻译、ASR、TTS 和部分图像能力统一到一个 Rust 生态内

### 4.2 `candle` 的定位

建议在 `deep_learning` crate 内把 `candle` 放在运行时层，而不是直接散落到各功能模块中。

也就是说：

1. `runtime.rs` 负责设备、dtype、模型载入和张量生命周期管理
2. 各能力模块只负责本任务的前处理、后处理和推理调用
3. UI 层不直接感知 `candle`

### 4.3 图像生成路线

首版建议仍然优先使用 `candle` 路线，但目标要收缩。

建议：

1. Rust 侧负责任务、参数、结果管理
2. 图像生成首版先只支持最小可运行路径
3. 不要求首版覆盖完整扩散模型生态能力

原因：

1. 图像生成模型体积大
2. 显存和运行环境要求高
3. 即便采用 `candle`，图像生成仍然是五个能力里最复杂的一项

### 4.4 音频相关能力

对于 `Whisper`、`人声分离`、`语音生成`，建议统一输出为标准音频文件，便于复用当前仓库已有音频播放能力。

### 4.5 当前模型选型总览

建议按“首版主模型 + 后续扩展模型”的方式规划，而不是每个能力一开始就绑定唯一模型。

| 能力 | 首版主推荐模型 | 后续扩展模型 | 说明 |
|------|----------------|--------------|------|
| 本地翻译 | `LLaMAX/GlotMAX-17-8B` | `luoyingfeng/BiMaTE-4B`、`patmodels/bpm` | 当前翻译路线固定采用 `Qwen` 系增强模型，`GlotMAX-17-8B` 作为质量优先主模型，其他模型降为备选 |
| 人声分离 | `htdemucs_ft` | 无 | 当前设备资源足够，直接采用质量优先的人声/伴奏二分离模型 |
| Whisper | `openai/whisper-large-v3` | `openai/whisper-base` | 当前设备 `16GB` 显存已经验证可跑 `large-v3`，因此直接以高质量模型作为默认方案 |
| 图像生成 | `stabilityai/sdxl-turbo` | `stabilityai/stable-diffusion-xl-base-1.0`、`stabilityai/stable-diffusion-xl-refiner-1.0` | `16GB` 显存已经足够跑 `SDXL` 级模型，首版更适合先做响应更快的 `SDXL-Turbo` |
| 语音生成 | `Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice` | `Qwen/Qwen3-TTS-12Hz-0.6B-CustomVoice`、`fishaudio/fish-speech-1.2-sft`、`GPT-SoVITS`、`Style-Bert-VITS2` | 当前需求只关注中文和日文，`Qwen3-TTS`、`Fish Speech`、`GPT-SoVITS`、`Style-Bert-VITS2` 都更贴近中日文场景 |

### 4.6 结合当前设备的模型建议

当前设备条件：

1. `NVIDIA RTX 4070 Ti Super`
2. 显存 `16 GB`

基于这个条件，首版更建议采用“默认模型 + 高质量模型”双层策略：

1. 本地翻译：
   - 默认：`LLaMAX/GlotMAX-17-8B`
   - 均衡备选：`luoyingfeng/BiMaTE-4B`
   - 领域特化备选：`patmodels/bpm`
   - 理由：既然当前翻译优先追求效果，并且设备有 `16GB` 显存，就直接把 `GlotMAX-17-8B` 作为主模型
2. Whisper：
   - 默认：`openai/whisper-large-v3`
   - 轻量回退：`openai/whisper-base`
   - 理由：当前设备已经验证可跑 `large-v3`，因此直接采用高质量默认模型
3. 图像生成：
   - 默认：`stabilityai/sdxl-turbo`
   - 高质量模式：`stabilityai/stable-diffusion-xl-base-1.0`
   - 理由：`SDXL` 在 11GB 左右环境已经有人尝试运行，16GB 更适合作为桌面默认硬件档位
4. 语音生成：
   - 默认：`Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice`
   - 轻量回退：`Qwen/Qwen3-TTS-12Hz-0.6B-CustomVoice`
   - 备选：`fishaudio/fish-speech-1.2-sft`
   - 特化备选：`GPT-SoVITS`、`Style-Bert-VITS2`
5. 人声分离：
   - 默认：`htdemucs_ft`

## 5. 整体架构

建议目录结构：

```text
deep_learning/
├── Cargo.toml
└── src/
    ├── lib.rs
    ├── error.rs
    ├── model.rs
    ├── runtime.rs
    ├── task.rs
    ├── output.rs
    ├── translation.rs
    ├── separation.rs
    ├── whisper.rs
    ├── image_generation.rs
    └── tts.rs
```

UI 侧建议新增：

```text
ui/src/homepage/deep_learning/
├── components.rs
├── plugin.rs
├── resources.rs
└── systems.rs
```

职责划分：

| 文件 | 职责 |
|------|------|
| `model.rs` | 统一模型元数据、模型版本、模型目录描述 |
| `runtime.rs` | 基于 `candle` 的推理运行时、设备选择、任务执行和取消 |
| `task.rs` | 各类任务请求和任务状态 |
| `output.rs` | 统一输出结果结构 |
| `translation.rs` | 本地翻译服务 |
| `separation.rs` | 人声分离服务 |
| `whisper.rs` | 语音识别服务 |
| `image_generation.rs` | 图像生成服务 |
| `tts.rs` | 语音生成服务 |

## 6. 核心数据结构

建议核心结构统一为任务驱动：

```rust
/// 深度学习任务类型。
pub enum DlTaskKind {
    /// 本地翻译任务。
    Translation,

    /// 人声分离任务。
    Separation,

    /// Whisper 语音识别任务。
    Whisper,

    /// 图像生成任务。
    ImageGeneration,

    /// 语音生成任务。
    SpeechSynthesis,
}

/// 深度学习任务唯一标识。
pub struct DlTaskId(pub u64);

/// 深度学习任务运行状态。
pub struct DlTaskStatus {
    /// 当前任务 ID。
    pub id: DlTaskId,

    /// 当前任务类型。
    pub kind: DlTaskKind,

    /// 当前任务进度，取值范围为 0.0 到 1.0。
    pub progress: f32,

    /// 当前任务状态描述文本。
    pub message: String,

    /// 当前任务是否已经结束。
    pub is_finished: bool,
}

/// 本地翻译请求。
pub struct TranslationRequest {
    /// 源语言代码，例如 `zh`、`en`。
    pub source_language: String,

    /// 目标语言代码，例如 `zh`、`en`。
    pub target_language: String,

    /// 待翻译的原始文本。
    pub text: String,
}

/// Whisper 语音识别请求。
pub struct WhisperRequest {
    /// 输入音频或视频文件路径。
    pub input_path: PathBuf,

    /// 可选的语言提示。
    pub language: Option<String>,

    /// 是否输出时间戳片段。
    pub with_timestamps: bool,
}

/// 语音生成请求。
pub struct SpeechSynthesisRequest {
    /// 待合成的文本内容。
    pub text: String,

    /// 说话人或音色标识。
    pub speaker: String,

    /// 语速倍率。
    pub speed: f32,
}
```

说明：

1. 首版统一通过任务系统触发推理
2. 所有能力都需要有可查询状态
3. 音频、文本、图片结果统一收敛到输出结构层

## 7. 各能力设计

### 7.1 本地翻译

目标：

- 首版只支持两条固定翻译方向：
  1. 英译中
  2. 日译中
- 不做中译英
- 不做多语种自由互译
- 首版重点是把固定目标语言为中文的体验做稳

推荐模型：

1. 主推荐：`LLaMAX/GlotMAX-17-8B`
2. 均衡备选：`luoyingfeng/BiMaTE-4B`
3. 领域特化备选：`patmodels/bpm`

模型链接：

1. `LLaMAX/GlotMAX-17-8B`
   - https://huggingface.co/LLaMAX/GlotMAX-17-8B
2. `luoyingfeng/BiMaTE-4B`
   - https://huggingface.co/luoyingfeng/BiMaTE-4B
3. `patmodels/bpm`
   - https://huggingface.co/patmodels/bpm

说明：

1. `GlotMAX-17-8B` 基于 `Qwen3` 路线做 translation-enhanced tuning，覆盖 `en`、`ja`、`zh` 在内的 `17` 种语言，和当前需求直接匹配
2. 你当前设备有 `RTX 4070 Ti Super 16GB`，既然已决定翻译效果优先，就可以直接把 `8B` 量级模型作为主方案
3. `BiMaTE-4B` 仍然适合作为更均衡、更轻量的备选
4. `patmodels/bpm` 更适合专利和技术文档场景，不作为默认通用模型

当前固定排序：

1. 默认模型：`LLaMAX/GlotMAX-17-8B`
2. 轻量备选：`luoyingfeng/BiMaTE-4B`
3. 专利/技术文本特化：`patmodels/bpm`

输入：

- 文本

输出：

- 翻译后的文本

支持语言方向：

1. `English -> Chinese`
2. `Japanese -> Chinese`

UI 建议：

1. 源文本输入框
2. 源语言和目标语言选择
3. 翻译按钮
4. 输出结果框

首版边界：

1. 先做短文本和中等长度文本
2. 长文本按段切分处理
3. 首版固定目标语言为中文

### 7.2 人声分离

目标：

- 首版支持二分离：
  1. 人声
  2. 伴奏
- 二期再评估四分离或更多 stem 输出

推荐模型：

1. 主推荐：`htdemucs_ft`

模型链接：

1. `htdemucs_ft`
   - 官方仓库：https://github.com/facebookresearch/demucs
   - 可直接下载的转换权重示例：https://huggingface.co/CrazeDigger/htdemucs

说明：

1. `Demucs` 系列直接面向音乐源分离，和“人声/伴奏分离”需求最贴合
2. `htdemucs_ft` 质量优先，适合正式结果输出

输入：

- `wav`
- `mp3`

输出：

- `vocals.wav`
- `accompaniment.wav`

UI 建议：

1. 文件选择
2. 开始分离按钮
3. 输出路径显示
4. 可选试听入口

和现有项目的契合点：

1. 可复用音频播放页做试听
2. 可复用统一输出目录管理

### 7.3 Whisper

目标：

- 本地多语言语音识别
- 支持长音频离线转写
- 支持语音翻译到英文
- 支持时间戳输出

推荐模型：

1. 主推荐：`openai/whisper-large-v3`
2. 开发调试备选：`openai/whisper-base`

模型链接：

1. `openai/whisper-large-v3`
   - https://huggingface.co/openai/whisper-large-v3
2. `openai/whisper-base`
   - https://huggingface.co/openai/whisper-base

说明：

1. `Whisper` 模型同时覆盖 ASR 和 speech translation
2. `whisper-large-v3` 支持 99 种语言
3. 当前设备 `RTX 4070 Ti Super 16GB` 已经验证可运行 `large-v3`
4. 因此首版直接以 `large-v3` 作为默认模型，不再额外做 `medium` 档

输入：

- 音频文件
- 视频文件抽取出的音轨

输出：

- 识别文本
- 可选时间戳片段
- 可选英文翻译文本

UI 建议：

1. 选择音频或视频文件
2. 语言设置
3. 是否输出时间戳
4. 结果文本区

和现有项目的契合点：

1. 可复用音频播放和视频播放模块作为输入来源
2. 后续可接报告模块导出识别结果

### 7.4 图像生成

目标：

- 首版支持标准文生图
- 默认优先保证交互响应速度
- 首版只支持两个固定输出尺寸：
  1. `1024 x 768`
  2. `1920 x 1080`
- 二期再考虑 base + refiner 双阶段生成

推荐模型：

1. 默认推荐：`stabilityai/sdxl-turbo`
2. 高质量模式：`stabilityai/stable-diffusion-xl-base-1.0`
3. 后续扩展：`stabilityai/stable-diffusion-xl-refiner-1.0`

模型链接：

1. `stabilityai/sdxl-turbo`
   - https://huggingface.co/stabilityai/sdxl-turbo
2. `stabilityai/stable-diffusion-xl-base-1.0`
   - https://huggingface.co/stabilityai/stable-diffusion-xl-base-1.0
3. `stabilityai/stable-diffusion-xl-refiner-1.0`
   - https://huggingface.co/stabilityai/stable-diffusion-xl-refiner-1.0

说明：

1. `SDXL-Turbo` 是 `SDXL 1.0` 的蒸馏版本，适合做快速预览和桌面交互
2. `SDXL base` 适合做质量优先输出
3. `11 GB` 左右环境已经有人尝试运行 `SDXL base`，当前 `16 GB` 显存更适合直接把 `SDXL` 路线作为首版目标

输入：

- Prompt
- 固定尺寸选择
- 步数
- 种子

输出：

- PNG 图片

首版建议：

1. Rust 侧只做参数、任务和结果管理
2. 首版优先采用 `SDXL-Turbo` 默认路径
3. 高质量输出时再切到 `SDXL base`
4. 先不追求复杂工作流，如 ControlNet、LoRA 组合和图生图
5. 尺寸只开放 `1024 x 768` 和 `1920 x 1080`

UI 建议：

1. Prompt 输入框
2. 固定尺寸选择：`1024 x 768` / `1920 x 1080`
3. 采样步数
4. 结果预览
5. 保存图片

### 7.5 语音生成

目标：

- 首版只考虑中文和日文语音生成
- 不要求首版覆盖英文或更多欧洲语言
- 除了基础 TTS，还希望支持以下至少一类能力：
  1. 预置说话人
  2. 风格控制
  3. 音色克隆
- 二期再扩展长文本和更强的克隆控制

推荐模型：

1. 主推荐：`Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice`
2. 轻量回退：`Qwen/Qwen3-TTS-12Hz-0.6B-CustomVoice`
3. 高质量备选：`fishaudio/fish-speech-1.2-sft`
4. 克隆和中日文特化备选：`GPT-SoVITS`
5. 日文情感表达特化备选：`Style-Bert-VITS2`

模型链接：

1. `Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice`
   - https://huggingface.co/Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice
2. `Qwen/Qwen3-TTS-12Hz-0.6B-CustomVoice`
   - https://huggingface.co/Qwen/Qwen3-TTS-12Hz-0.6B-CustomVoice
3. `fishaudio/fish-speech-1.2-sft`
   - https://huggingface.co/fishaudio/fish-speech-1.2-sft
4. `GPT-SoVITS`
   - https://github.com/RVC-Boss/GPT-SoVITS
5. `Style-Bert-VITS2`
   - https://github.com/litagin02/Style-Bert-VITS2

说明：

1. `Qwen3-TTS` 官方仓库明确支持稳定语音生成、自由音色设计和音色克隆，并给出 `0.6B` / `1.7B` 两档模型
2. `Qwen3-TTS` 的官方指标页明确列出中文和日文表现，也支持自定义说话人和 voice design
3. `Fish Speech 1.2` 明确以中文、日文、英文为主训练，适合作为高质量备选
4. `GPT-SoVITS` 明确支持中文、日文、英文的 zero-shot / few-shot TTS，更适合做少样本克隆路线
5. `Style-Bert-VITS2` 在日语情感表达和固定角色音色上更有针对性，适合作为日文特化方案

工程建议：

1. 首版最值得优先验证的是 `Qwen3-TTS-12Hz-1.7B-CustomVoice`
2. 如果更重视少样本克隆和中文社区现成资料，`GPT-SoVITS` 更适合作为实验路线
3. 如果更重视日文情感和固定角色音色，`Style-Bert-VITS2` 更适合作为特化路线

输入：

- 文本
- 音色
- 语速

输出：

- 音频文件

支持语言：

1. 中文
2. 日文

UI 建议：

1. 文本输入框
2. 音色选择
3. 语速设置
4. 生成按钮
5. 播放试听

和现有项目的契合点：

1. 可复用音频播放模块试听生成结果

## 8. 输出目录建议

不要和报告、截图输出混在一起，建议单独新增：

```text
deep_learning_output/
├── translation/
├── separation/
├── whisper/
├── image_generation/
└── tts/
```

说明：

1. 统一由 `deep_learning` crate 管理输出路径
2. 没有目录时自动创建
3. UI 侧只需要展示结果，不直接处理路径规则

模型目录建议：

```text
deepl_models/
├── translation/
├── separation/
├── whisper/
├── image_generation/
└── tts/
```

说明：

1. 模型文件统一放在项目根目录 `deepl_models/`

## 12. 风险与应对

### 12.1 模型体积大

风险：

- 模型下载和分发成本高

应对：

1. 首版先支持手动放置到 `deepl_models/`
2. 模型管理层统一校验版本和路径

### 12.2 推理耗时长

风险：

- 如果把推理直接放在主线程，UI 会卡死

应对：

1. 所有推理任务走后台异步执行
2. 后台任务通过 Bevy `Message` 回传状态和结果
3. UI 在主线程中消费 `Message` 并刷新状态资源

### 12.3 模型接入方式不统一

风险：

- 五个能力如果各自选择不同模型接入方式，后续维护成本会很高

应对：

1. 优先统一任务结构
2. 优先统一输出目录和错误模型
3. `candle` 相关细节放到运行时和服务实现内部，不暴露给 UI

### 12.4 图像生成复杂度最高

风险：

- 图像生成对设备、显存和依赖要求高，最容易拖慢整体进度

应对：

1. 最后接入图像生成
2. 即便采用 `candle`，首版也只做最小闭环，不追求一步到位覆盖完整工作流

## 13. 最终结论

深度学习模块建议落成一个独立的 `deep_learning` crate，首版先聚焦：

1. 统一推理运行时
2. 深度学习测试页
3. Whisper
4. 语音生成
5. 本地翻译

后续再逐步扩展：

1. 人声分离
2. 图像生成

这样做可以先把“统一接入能力”和“UI 可调用能力”做稳，再逐步增加更重的模型能力。
