# 视频播放设计方案

## 1. 概述

视频播放功能用于在当前 `Bevy 0.18 + ECS + Plugin` 桌面应用中加载和播放本地视频文件，首版聚焦 `mp4(H.264 + AAC)`，覆盖以下能力：

1. 读取本地 `mp4`
2. 播放 / 暂停
3. 独立弹窗显示
4. 同时播放多个视频

首版目标不是做完整媒体播放器，而是先完成一个稳定、可扩展的最小闭环：

1. 能稳定解码 `mp4`
2. 能把视频帧更新到 Bevy 纹理
3. 能接入 GStreamer 的音视频同步播放能力
4. 能为后续主窗口与单个独立弹窗、多实例播放保留清晰边界

## 2. 库选型结论

视频播放功能固定采用 `gstreamer-rs` 生态，不再保留其他主方案。

推荐依赖组合：

- `gstreamer`
- `gstreamer-app`
- `gstreamer-video`

选择理由：

1. `gstreamer-rs` 由 GStreamer 官方组织维护，生态成熟
2. 当前需求包含音视频同步、播放状态、独立弹窗、多实例并发，适合直接建立在播放器内核之上
3. 可以通过 `appsink` 把视频帧拉回 Rust，再交给 Bevy 负责 UI 和纹理显示

## 3. 为什么优先推荐 GStreamer

### 3.1 和当前需求更匹配

当前不是只要“从 mp4 拿到一帧图像”，而是需要：

- 有音频
- 有播放状态
- 有最多一个独立弹窗
- 最多同时播放两个视频

### 3.2 GStreamer 的实际落点

本项目不建议把 GStreamer 直接当“黑盒窗口播放器”，而是把它当媒体内核：

1. GStreamer 负责解复用、解码、音频输出、播放状态机、时钟
2. `appsink` 负责把解码后的视频帧拉回 Rust
3. Rust 侧把帧转成 Bevy `Image`
4. Bevy 负责 UI、交互、窗口和纹理显示

这样做的好处：

- 保留 Bevy 作为统一 UI 框架
- 不需要把视频窗口完全托管给外部播放器控件
- 后续做截图、叠加标记、联动控制会更灵活

## 4. 整体架构

建议新增一个内部 crate，专门负责媒体播放内核：

```text
media_player/
├── Cargo.toml
└── src/
    ├── lib.rs
    ├── player.rs
    ├── pipeline.rs
    ├── events.rs
    ├── frame.rs
    ├── audio.rs
    └── error.rs

ui/src/homepage/video_player/
├── components.rs
├── resources.rs
├── systems.rs
└── plugin.rs
```

分层职责：

| 层级 | 职责 |
|------|------|
| `media_player` crate | 媒体文件加载、GStreamer pipeline、播放控制、帧输出 |
| `ui::homepage::video_player` | 页面生命周期、列表/控制栏、纹理刷新、用户交互 |
| Bevy 窗口层 | 主窗口和最多一个独立弹窗管理 |

## 5. 核心数据结构

### 5.1 播放器资源

```rust
#[derive(Resource, Default)]
pub struct VideoPlaybackState {
    /// 主窗口播放器实例
    pub main_player: Option<VideoPlayerEntry>,
    /// 弹窗播放器实例
    pub popup_player: Option<VideoPlayerEntry>,
}

pub struct VideoPlayerEntry {
    pub id: VideoPlayerId,
    pub source: std::path::PathBuf,
    pub status: PlaybackStatus,
    pub duration_ms: Option<u64>,
    pub position_ms: u64,
    pub target_window: VideoWindowTarget,
    pub latest_frame: Option<VideoFrame>,
}

pub enum PlaybackStatus {
    Idle,
    Loading,
    Playing,
    Paused,
    Ended,
    Error,
}

pub enum VideoWindowTarget {
    Embedded,
    Detached(bevy::window::WindowId),
}
```

### 5.2 视频帧

```rust
pub struct VideoFrame {
    pub width: u32,
    pub height: u32,
    /// 统一转成 RGBA8，便于直接上传 Bevy Image
    pub pixels_rgba: Vec<u8>,
    /// 播放时间戳，单位纳秒
    pub pts_ns: u64,
}
```

首版统一转 `RGBA8`，理由很直接：

- 方便上传到 Bevy 纹理
- UI 层逻辑最简单
- 后续要做截图和叠加绘制也方便

代价是 CPU 上多一次像素格式转换，但首版可接受。

## 6. GStreamer 方案设计

### 6.1 推荐依赖

`media_player` crate 建议优先加入：

| 依赖 | 用途 |
|------|------|
| `gstreamer` | GStreamer 核心能力 |
| `gstreamer-app` | `appsink` 拉取视频帧 |
| `gstreamer-video` | 视频格式与帧信息处理 |
| `crossbeam-channel` 或 `flume` | 播放线程与 Bevy 主线程通信 |
| `thiserror` | 统一错误定义 |

### 6.2 Pipeline 思路

首版建议直接走：

```text
filesrc -> decodebin -> videoconvert -> appsink
                      -> audioconvert -> audioresample -> autoaudiosink
```

说明：

- `filesrc` 读取本地文件
- `decodebin` 自动识别 `mp4` 中音视频流并解码
- 视频分支接 `appsink`，由 Rust 主动取帧
- 音频分支接系统音频输出

首版尽量不要自己在 Rust 侧播放音频，否则会把音频缓冲、时钟同步、输出设备切换问题一起带进来。

### 6.3 播放控制

`GStreamer` 原生支持：

- `Playing`
- `Paused`
- `Ready`
- `Null`

同时可通过 `seek` 实现调整位置。

因此首版的播放/暂停不建议自己造状态机，只需要在 Rust 侧封装成项目自己的命令接口。

## 7. 与 Bevy 的集成方式

### 7.1 主线程和播放线程拆分

建议拆成两条链路：

1. 播放线程
2. Bevy 主线程

播放线程负责：

- 初始化 GStreamer
- 创建 pipeline
- 拉取视频帧
- 接收播放命令
- 监听 bus 错误和结束事件

Bevy 主线程负责：

- 页面和控件
- 把主窗口播放器的最新帧写入主窗口 `Assets<Image>`
- 把弹窗播放器的最新帧写入弹窗 `Assets<Image>`

### 7.2 帧更新策略

首版建议使用“仅保留最新帧”策略，而不是缓存完整队列：

- 播放线程持续产生帧
- 每个播放器实例只保留一张最新帧
- Bevy 每帧读取一次并刷新纹理

这样做的原因：

- 逻辑最简单
- 不容易因为 UI 卡顿堆积大量帧
- 对播放页这种“看当前画面”的需求已经够用

缺点是极端情况下可能丢帧，但这比内存堆积更可控。

## 8. 独立弹窗方案

### 8.1 推荐做法

独立弹窗不新建独立进程，但会新建一个独立播放器实例。首版建议只支持“主窗口 + 最多一个独立弹窗”，也就是最多同时播放两个视频：

1. 主窗口视频一个播放器实例，对应一个解码 pipeline
2. 弹窗视频一个播放器实例，对应另一个解码 pipeline
3. 当用户点击“独立弹窗显示”时，先打开一个独立播放器弹窗，再在弹窗内选择视频文件并独立播放

这样做的边界很清晰：

- 主窗口和弹窗各自加载自己的文件
- 主窗口和弹窗各自维护自己的播放状态
- 主窗口和弹窗各自输出自己的音频和视频

### 8.2 窗口管理

Bevy 侧建议：

- 最多只创建一个独立视频 `Window`
- 主窗口固定承载主窗口视频实例
- 弹窗固定承载弹窗视频实例
- 建立当前弹窗与 `VideoPlayerId` 的映射
- 关闭窗口时只回收弹窗，不自动销毁播放器，除非用户选择关闭媒体

## 9. 多视频并发方案

### 9.1 首版边界

“同时播放多个视频”建议首版固定做成：

- 最多 2 个并发实例
- 主窗口 1 个，弹窗 1 个
- 每个实例一个独立 pipeline

### 9.2 资源控制

建议增加保护策略：

- 限制最大并发播放器数量
- 限制单文件分辨率，例如先只保证 `1080p`
- 超出上限时给出明确错误提示

```text

+--------------------------------------------------------------------------------+
| [打开文件] [播放/暂停] [独立弹窗] [关闭视频]                                  |
+--------------------------------------------------------------------------------+
|                                                                                |
|  +--------------------------------------------------------------------------+  |
|  |                             主窗口视频显示区                              |  |
|  |                                                                          |  |
|  |                                画面纹理                                  |  |
|  |                                                                          |  |

```

交互约束：

- 主窗口界面只保留一个视频显示区域

- 用户点击“独立弹窗”后，先打开弹窗
- 弹窗中的视频文件由弹窗自己选择，不复用主窗口当前文件
- 主窗口和弹窗各自独立播放、暂停和关闭

|  |                       弹窗视频显示区                       |  |

- 弹窗关闭后，只销毁弹窗播放器，不影响主窗口播放器

## 12. 风险与应对

### 12.1 GStreamer 部署复杂

风险：

- Windows 下需要安装匹配工具链的 GStreamer Runtime / Development
- CI 和用户环境需要额外准备动态库

应对：

- 首版先只承诺 Windows 开发环境
- 在文档里明确 GStreamer 安装步骤
- 后续再评估是否随安装包一起分发运行时

### 12.2 视频帧颜色转换开销

风险：

- 转成 `RGBA8` 会增加 CPU 开销

应对：

- 首版先换开发效率
- 后续如果性能不足，再考虑直接处理 `BGRA` 或 GPU 侧转换

### 12.3 多实例 CPU / 内存压力

风险：

- 多路 `1080p` 解码会明显增加资源占用

应对：

- 限制最大并发实例数
- 默认只保留最新帧
- 需要时增加后台暂停策略

## 13. 最终结论

建议本项目的视频播放首版采用：

- 媒体内核：`GStreamer`
- Rust 绑定：`gstreamer-rs`
- 实际依赖：`gstreamer` / `gstreamer-app` / `gstreamer-video`
- UI 与窗口：`Bevy 0.18`
- 数据交换：播放器线程输出最新 `RGBA8` 帧，Bevy 负责纹理显示

这条路线最符合当前需求列表，尤其是：

- `mp4` 读取
- 播放 / 暂停
- 独立弹窗
- 多视频并发
