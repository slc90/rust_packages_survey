# 报告设计方案

## 1. 概述

报告模块用于把当前项目中的分析结果、图表和基础信息导出成可交付文档。

根据 `docs/requirements.md`，当前范围只有两项：

1. 导出 Word
2. 导出 PDF

首版目标不做复杂模板系统，而是先完成一个稳定、可复用的报告导出闭环：

1. 能组织报告内容
2. 能插入文字、标题、表格、图片
3. 能嵌入项目里生成的图表
4. 能分别导出 `.docx` 和 `.pdf`

## 2. 设计结论

建议新增独立 crate：

- `report_generator`

这个 crate 只负责：

- 报告数据模型
- 报告章节组织
- Word/PDF 导出
- 图表图片嵌入

不负责：

- Bevy 页面 UI
- 页面状态管理
- 算法计算本身

也就是说，报告模块是“结果输出层”，输入来自：

1. `signal_processing`
2. `medical_image`
3. 后续其他分析模块

## 3. 库选型建议

首版建议选型如下：

- Word：`docx-rs`
- PDF：`printpdf`
- 图表出图：复用现有 `plotters`

### 3.1 Word 导出

推荐：

- `docx-rs`

原因：

1. 当前版本较新，`docs.rs` 上 `0.4.x` 仍在持续更新
2. 专门用于 `.docx` 写入，定位清晰
3. API 已经覆盖段落、文本、表格、图片等常见报告需求
4. 纯 Rust 路线，适合当前项目

来源：

- `docx-rs` docs.rs: https://docs.rs/crate/docx-rs/latest
- `docx-rs` lib.rs 页面: https://lib.rs/crates/docx-rs

### 3.2 PDF 导出

推荐：

- `printpdf`

原因：

1. 当前版本较新，仍在持续维护
2. 支持 PDF 创建、字体、图形、图片和基础布局
3. 比纯底层 PDF writer 更适合直接生成报告

要注意的边界：

1. `printpdf` 不是高层 Word 式排版系统
2. 复杂页面布局仍然需要我们自己封装一层

来源：

- `printpdf` docs.rs: https://docs.rs/crate/printpdf/latest

### 3.3 不作为首选的库

#### `genpdf`

优点：

1. 更偏高层 PDF 生成
2. 纯 Rust

问题：

1. 当前最新发布距离现在较久
2. 活跃度和演进速度不如 `printpdf`

来源：

- `genpdf` docs.rs: https://docs.rs/crate/genpdf/latest

#### `lopdf`

优点：

1. 很成熟
2. 适合 PDF 操作、修改、合并

问题：

1. 更偏底层 PDF 文档操作
2. 不适合作为首版“直接生成业务报告”的主入口

来源：

- `lopdf` docs.rs: https://docs.rs/crate/lopdf/latest

#### `pdf-writer`

优点：

1. 现代、清晰、底层控制强

问题：

1. 太偏底层
2. 首版报告导出会增加过多版面组织工作

来源：

- `pdf-writer` docs.rs: https://docs.rs/crate/pdf-writer/latest

## 4. 整体架构

建议目录结构：

```text
report_generator/
├── Cargo.toml
├── examples/
│   ├── export_docx.rs
│   └── export_pdf.rs
└── src/
    ├── lib.rs
    ├── error.rs
    ├── model.rs
    ├── image.rs
    ├── docx.rs
    └── pdf.rs
```

职责划分：

| 文件 | 职责 |
|------|------|
| `model.rs` | 统一报告数据结构 |
| `image.rs` | 图表图片读取与尺寸适配 |
| `docx.rs` | `docx-rs` 导出实现 |
| `pdf.rs` | `printpdf` 导出实现 |
| `error.rs` | 统一错误类型 |
| `examples/` | 人工验收和导出样例 |

## 5. 核心数据结构

建议先做一套与导出库无关的报告模型：

```rust
pub struct ReportDocument {
    pub title: String,
    pub subtitle: Option<String>,
    pub sections: Vec<ReportSection>,
}

pub struct ReportSection {
    pub heading: String,
    pub blocks: Vec<ReportBlock>,
}

pub enum ReportBlock {
    Paragraph(String),
    KeyValueTable(Vec<(String, String)>),
    Image(ReportImage),
}

pub struct ReportImage {
    pub path: std::path::PathBuf,
    pub caption: Option<String>,
    pub width_px: Option<u32>,
}
```

这样做的目的很明确：

1. 不让 `docx-rs` 和 `printpdf` 的类型泄漏到上层
2. 同一份报告数据可以输出为 Word 和 PDF
3. 后续如果替换底层导出库，影响面更小

## 6. 图表嵌入策略

报告中最关键的非文本内容，实际上是图表和截图。

建议统一策略：

1. 图表先由 `plotters` 输出成 PNG
2. 报告模块只负责读取 PNG 并嵌入

不要首版就做：

1. 在报告导出过程中临时现场画图
2. 让 `docx-rs` 和 `printpdf` 分别拥有两套独立画图逻辑

这样更稳，也更容易复用现有 `signal_processing/examples` 的出图结果。

## 7. Word 导出设计

首版 `docx-rs` 路线建议支持：

1. 文档标题
2. 二级标题
3. 普通段落
4. 简单表格
5. 插入 PNG 图片

首版不急着做：

1. 页眉页脚
2. 目录自动生成
3. 复杂样式模板
4. 页码控制和分节控制

因为对当前项目来说，更重要的是“能稳定产出报告文件”，不是先把 Word 排版做到出版级。

## 8. PDF 导出设计

首版 `printpdf` 路线建议支持：

1. 标题和段落文字
2. 图片插入
3. 基础分页
4. 简单表格或键值对块

首版不建议做：

1. 复杂自动排版引擎
2. HTML 转 PDF
3. 完整模板 DSL

更务实的路线是：

1. 先做固定样式页面
2. 文本和图表顺序排版
3. 内容过长时自动换页

## 9. Example 说明

建议 `report_generator/examples/` 至少保留两个例子：

### 9.1 export_docx.rs

用途：

- 生成一个最小 `.docx` 报告

内容建议：

1. 标题
2. 一段简介文字
3. 一个键值表格
4. 一张频谱图图片

### 9.2 export_pdf.rs

用途：

- 生成一个最小 `.pdf` 报告

内容建议：

1. 标题
2. 一段简介文字
3. 一张频谱图图片
4. 一段结论文字

建议直接复用现有 `signal_processing_output/fft_demo.png` 这类图片作为示例输入。

## 10. 实现步骤

### Phase 1: 骨架

1. 新增 `report_generator` crate 并加入 workspace
2. 加入 `docx-rs`、`printpdf`
3. 新增 `lib.rs`、`error.rs`、`model.rs`
4. 定义统一报告数据结构

### Phase 2: Word 导出

1. 实现 `docx.rs`
2. 支持标题、段落、表格、图片
3. 新增 `examples/export_docx.rs`
4. 验证 `.docx` 可正常打开

### Phase 3: PDF 导出

1. 实现 `pdf.rs`
2. 支持标题、段落、图片和分页
3. 新增 `examples/export_pdf.rs`
4. 验证 `.pdf` 可正常打开

### Phase 4: 工程化

1. 补单元测试和 example 运行验证
2. 增加错误分类
3. 回写 `structure.md`
4. 回写 `docs/requirements.md`

## 11. 风险与应对

### 11.1 PDF 排版复杂度偏高

风险：

- `printpdf` 更接近文档绘制，不是现成的高层报告模板系统

应对：

- 首版只做固定样式
- 先支持最核心的标题、段落、图片、分页

### 11.2 Word 和 PDF 双导出导致模型失控

风险：

- 如果一开始就让两种导出各自长出独立数据结构，后面会难维护

应对：

- 统一先建 `ReportDocument`
- Word/PDF 都只做渲染层

### 11.3 图片尺寸和版面适配问题

风险：

- 图表图片尺寸不统一时，导出效果容易失控

应对：

- 统一报告图片输入规范
- 先约束为 PNG
- 首版只支持少量固定尺寸策略

## 12. 最终结论

报告模块建议新增独立的 `report_generator` crate，首版路线收敛为：

1. `docx-rs` 负责 Word 导出
2. `printpdf` 负责 PDF 导出
3. `plotters` 继续负责图表出图
4. 统一 `ReportDocument` 数据模型
5. 先用 `examples/` 验证导出，不急着接 UI
