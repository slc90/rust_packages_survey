# 范式设计方案

## 1. 概述

范式模块用于在当前 `Bevy` 桌面程序中提供“刺激呈现 + 播放控制 + 显示器输出”的统一能力。

根据 `docs/requirements.md`，当前范式部分首版目标包含：

1. `gif` 预览
2. 选定显示器全屏播放
3. `P300`
4. 暂停 / 播放
5. 退出播放

首版目标不是一次性做完整脑机接口实验平台，而是先完成一个稳定、可扩展的最小闭环：

1. 能在主窗口范式页面中自动显示 `gif` 素材
2. 能把真正的实验刺激内容投到用户选择的显示器全屏窗口
3. 能在主窗口侧控制开始、暂停、恢复和退出
4. 能支持首个实验范式 `P300`

## 2. 设计结论

建议新增独立 crate：

- `paradigm`

同时在 `ui/src/homepage/` 下新增独立页面模块：

- `ui/src/homepage/paradigm/`

职责边界建议如下：

`paradigm` crate 负责：

- 范式任务和配置数据结构
- 素材加载与播放时序
- `gif` 帧解码与缓存
- 目标显示器选择
- 播放状态机
- `P300` 刺激序列生成

UI 模块负责：

- 页面布局
- 参数输入
- 文件选择
- 播放控制按钮
- 状态显示

不建议把素材解码、实验时序和窗口控制全部堆在 UI 系统里，否则后续扩展新范式时会很快失控。

## 3. 总体路线

首版建议采用“主窗口素材显示 + 统一播放会话 + 分范式配置”的架构。

建议优先级：

1. `gif` 预览
2. 选定显示器全屏播放
3. 播放 / 暂停 / 退出
4. `P300`

原因：

1. 主窗口素材自动显示最容易先形成页面可见闭环
2. 目标显示器播放是后续所有实验刺激范式的共同底座
3. 播放控制是实验可用性的基础
4. `P300` 建立在目标显示器刺激显示和时序控制稳定之后再接入更稳

## 4. 功能目标与边界

### 4.1 `gif` 素材显示

目标：

1. 在主窗口范式页面中自动显示 `gif` 动图素材
3. 显示基础信息：
   - 文件名
   - 尺寸
   - 总帧数
    - 建议播放时长

首版边界：

1. 只要求支持单个 `gif`
2. 不做素材库管理
3. 不做拖拽导入
4. `gif` 不承担副屏正式播放内容
5. `gif` 不算范式类型，只是主窗口页面中的自动显示素材
6. 首版要求 `gif` 能正常动起来，而不是只显示首帧

### 4.2 选定显示器全屏播放

目标：

1. 自动识别当前系统显示器列表
2. 允许用户选择目标显示器
3. 在目标显示器创建无边框全屏播放窗口
4. 播放窗口仅显示实验刺激内容，不显示调试 UI
5. 播放内容由具体范式生成，例如 `P300` 刺激矩阵，而不是主窗口里的 `gif` 素材显示
6. 目标显示器可以和主窗口位于同一块屏幕，不强制要求双屏

首版边界：

1. 首版只支持单个全屏播放窗口
2. 允许只存在一个显示器的场景
3. 不做跨机器同步播放

实现约束：

1. 播放窗口直接采用 `Bevy` 官方多窗口能力实现
2. 全屏模式采用 `WindowMode::BorderlessFullscreen(MonitorSelection::...)`
3. 不单独绕开 `Bevy` 的窗口系统自己管理原生窗口

### 4.3 `P300`

目标：

1. 提供一个最小可运行的 `P300` 刺激范式
2. 刺激矩阵固定为 `6 x 6`
3. 支持行列闪烁
4. 支持目标字符配置
5. 支持单次 block 的播放和重复 block 播放

首版边界：

1. 只做视觉刺激呈现，不负责脑电采集
2. 不做在线分类
3. 不做实验结果统计
4. 不做多种 `P300` 变体切换

实现约束：

1. `P300` 刺激界面由 `Bevy` 在运行时直接绘制
2. 不依赖外部图片素材或预渲染帧序列
3. 字符矩阵、高亮效果和闪烁状态都由程序生成

### 4.4 播放控制

目标：

1. 主窗口可发起开始播放
2. 主窗口可暂停和恢复播放
3. 主窗口可主动退出播放
4. 播放状态可同步回 UI

状态建议固定为：

1. `Idle`
2. `Preparing`
3. `Playing`
4. `Paused`
5. `Finished`
6. `Error`

## 5. 架构设计

建议目录结构：

```text
paradigm/
├── Cargo.toml
└── src/
    ├── lib.rs
    ├── error.rs
    ├── session.rs
    ├── monitor.rs
    ├── gif.rs
    ├── player.rs
    ├── p300.rs
    └── output.rs
```

UI 侧建议新增：

```text
ui/src/homepage/paradigm/
├── components.rs
├── plugin.rs
├── resources.rs
└── systems.rs
```

职责划分：

| 文件 | 职责 |
|------|------|
| `error.rs` | 统一错误类型 |
| `session.rs` | 范式播放会话、状态和控制命令 |
| `monitor.rs` | 显示器枚举、目标显示器选择和校验 |
| `gif.rs` | `gif` 解码、帧缓存和预览元信息 |
| `player.rs` | 播放调度、暂停恢复、退出和时钟推进 |
| `p300.rs` | `P300` 刺激矩阵、行列随机序列和 block 配置 |
| `output.rs` | 播放结果、运行摘要和日志导出结构 |

## 6. 核心数据结构

建议统一围绕播放会话组织：

```rust
/// 范式类型。
pub enum ParadigmKind {
    /// P300 视觉刺激。
    P300,
}

/// 播放状态。
pub enum ParadigmPlaybackState {
    /// 空闲。
    Idle,

    /// 准备中。
    Preparing,

    /// 播放中。
    Playing,

    /// 已暂停。
    Paused,

    /// 已完成。
    Finished,

    /// 播放失败。
    Error,
}

/// 范式播放会话。
pub struct ParadigmSession {
    /// 当前会话唯一标识。
    pub id: u64,

    /// 当前范式类型。
    pub kind: ParadigmKind,

    /// 当前播放状态。
    pub state: ParadigmPlaybackState,

    /// 目标显示器索引。
    pub target_monitor_index: usize,
}

/// P300 播放请求。
pub struct P300Request {
    /// 目标字符。
    pub target_symbol: String,

    /// 每个 flash 持续时长，单位毫秒。
    pub flash_duration_ms: u64,

    /// flash 间隔时长，单位毫秒。
    pub inter_stimulus_interval_ms: u64,

    /// block 重复次数。
    pub block_count: u32,
}
```

说明：

1. 首版要统一主窗口控制逻辑，但只有真正进入实验播放的范式才进入全屏播放状态机
2. `gif` 只是主窗口页面中的素材显示，不进入范式枚举，也不产生独立播放会话
3. 主窗口 UI 只控制会话，不直接操作副屏窗口内部细节
4. 所有播放行为都要有可回写的状态文本

## 7. 关键设计

### 7.1 显示器选择

显示器播放不建议写死“第二块屏幕”，而是应当显式枚举显示器并提供选择。

建议策略：

1. 启动页面时读取显示器列表
2. 默认选择当前主窗口所在显示器
3. 用户可以主动切换到任意其他显示器
4. 进入播放前再次校验目标显示器仍然存在

### 7.2 全屏播放窗口

建议采用单独 `Window` 实体承载刺激内容，特点如下：

1. 无边框
2. 全屏
3. 黑色背景
4. 不承载交互控件
5. 只由主窗口通过消息或资源控制

这样既可以支持主窗口和播放窗口分离，也可以支持把播放窗口开在与主窗口相同的显示器上。

实现建议直接采用 `Bevy` 官方多窗口方案：

1. 主窗口保留当前控制页面
2. 范式播放时动态创建独立的播放 `Window` 实体
3. 该窗口的 `mode` 设置为 `WindowMode::BorderlessFullscreen(MonitorSelection::...)`
4. `MonitorSelection` 由页面中的目标显示器选择结果决定
5. 为该窗口单独创建相机
6. 范式刺激实体只渲染到该播放窗口，不复用主窗口内容树

这样做的原因：

1. 方案和 `Bevy` 官方窗口模型一致，升级时风险更低
2. 单显示器和多显示器场景都能统一处理
3. 播放窗口生命周期可以和范式会话严格绑定，退出时直接销毁窗口即可

建议新增播放窗口资源：

```rust
/// 范式播放窗口状态。
pub struct ParadigmPresentationWindowState {
    /// 播放窗口实体。
    pub window_entity: Option<Entity>,

    /// 当前目标显示器索引。
    pub target_monitor_index: usize,

    /// 当前窗口是否已经创建。
    pub is_open: bool,
}
```

建议创建窗口时的实现方向：

```rust
commands.spawn(Window {
    title: "Paradigm Presentation".to_string(),
    mode: WindowMode::BorderlessFullscreen(MonitorSelection::Primary),
    decorations: false,
    focused: true,
    ..default()
});
```

说明：

1. 这里的 `MonitorSelection::Primary` 只是示意，实际应替换为页面选中的目标显示器
2. 真正落地时需要把窗口实体、播放相机实体和会话状态一起保存，便于暂停、恢复和退出时统一回收

### 7.3 `gif` 素材显示

首版建议直接采用 `bevy_easy_gif` 方案完成主窗口 `gif` 动图显示，而不是自己先手写一套帧解码和播放时钟。

原因：

1. `gif` 有逐帧时长信息，不能简单按固定帧率更新
2. 当前目标只是让主窗口页面中的素材稳定动起来，不值得首版就自己维护完整 `gif` 播放器
3. `bevy_easy_gif` 已经专门处理了按帧时序切换 `Sprite.image` 的问题
4. 主窗口素材显示和实验刺激应分开，避免把素材展示误当成正式刺激输出

首版建议：

1. 在 `paradigm` crate 或 `ui` crate 中引入 `bevy_easy_gif`
2. 在范式页面进入时，从约定路径加载默认 `gif`
3. 通过 `Gif` 组件生成一个随时间自动切帧的预览实体
4. 预览实体只挂在主窗口范式页面，不进入全屏播放窗口
5. 页面退出时清理对应实体，避免后台继续播放

建议实现路径：

1. 在 `Cargo.toml` 中增加 `bevy_easy_gif`
2. 在主页或范式插件中注册 `bevy_easy_gif::GifPlugin`
3. 在 `ui/src/homepage/paradigm/systems.rs` 的 `on_enter` 中：
   1. 解析默认 `gif` 路径
   2. 生成用于预览的 `Node` / `ImageNode` 容器
   3. 生成 `Gif` 组件实体并附着到页面预览区域
4. 在 `resources.rs` 中记录当前 `gif` 的路径、名称和基础元信息
5. 在 `on_exit` 中清理 `gif` 预览实体

建议新增页面级状态：

```rust
/// 范式页面中的 GIF 预览状态。
pub struct ParadigmGifPreviewState {
    /// GIF 文件路径。
    pub path: PathBuf,

    /// 文件名称。
    pub file_name: String,

    /// 是否成功完成加载。
    pub is_loaded: bool,

    /// 当前状态说明。
    pub status_text: String,
}
```

建议页面生成逻辑：

```rust
commands.spawn((
    ParadigmGifPreviewMarker,
    Gif::from_asset(asset_server.load("embedded://embedded_assets/../assets/paradigm/default.gif")),
));
```

说明：

1. 这里的代码只是表达实现方向，具体 API 以接入时所用 `bevy_easy_gif` 版本为准
2. 如果后续发现插件和当前 `Bevy` 版本存在兼容问题，再退回到“手动解码帧 + 定时切换 `ImageNode.image`”的备用方案

备用方案建议：

1. 使用 `image` crate 解析 `gif`
2. 把每一帧转成 `bevy::image::Image`
3. 记录每帧 delay
4. 在 `Update` 系统中根据 `Time` 推进当前帧索引
5. 更新主窗口预览节点使用的 `Handle<Image>`

这个备用方案只用于插件不兼容时兜底，不建议作为首版默认实现。

### 7.4 `P300` 刺激设计

`P300` 首版建议采用经典行列闪烁矩阵，不要一开始就扩展到复杂变体。

推荐固定方案：

1. `6 x 6` 字符矩阵
2. 每轮包含 `6` 次行闪烁和 `6` 次列闪烁
3. 每轮顺序随机打乱
4. 目标字符高亮仅用于实验准备阶段，不在正式闪烁阶段持续显示

素材来源建议：

1. `P300` 不使用外部图片作为正式刺激素材
2. 播放窗口中的网格、字符、边框、背景和高亮效果都由 `Bevy UI` 或 `Sprite` 运行时直接绘制
3. 真正需要保存和管理的不是图片文件，而是：
   1. 字符矩阵数据
   2. 颜色方案
   3. 字号和布局参数
   4. 行列刺激序列

这样做的原因：

1. `P300` 的界面结构规则固定，非常适合程序化生成
2. 直接绘制比维护一批外部素材更容易统一尺寸、颜色和布局
3. 后续如果要调整字符集、网格尺寸或高亮样式，修改成本更低

建议绘制组成：

1. 一个全屏背景层
2. 一个居中的 `6 x 6` 网格容器
3. `36` 个字符单元
4. 当前高亮行或列的颜色/亮度切换
5. 可选的准备态目标字符提示

首版参数建议：

1. `flash_duration_ms = 100`
2. `inter_stimulus_interval_ms = 75`
3. `block_count = 10`

这组参数先作为工程默认值，后续再根据实验效果和显示器刷新率微调。

### 7.5 播放控制

播放控制建议统一进入 `player.rs` 中的命令接口，不要让多个 UI 系统分别改状态。

建议命令集合：

1. `Start`
2. `Pause`
3. `Resume`
4. `Stop`

建议行为约束：

1. `Paused` 状态下保留当前帧和当前刺激索引
2. `Resume` 从暂停点继续推进
3. `Stop` 后销毁全屏播放窗口并释放会话资源
4. 主窗口关闭页面时自动触发 `Stop`

### 7.6 `FixedUpdate` 与时序精度

范式模块确实需要较高时序稳定性，但不建议把“精确显示”简单等同为“全部放到 `FixedUpdate` 里”。

建议结论：

1. 使用 `FixedUpdate` 驱动范式逻辑状态机
2. 使用正常渲染帧负责把当前刺激状态显示到播放窗口
3. 不把 `FixedUpdate` 当成显示器真实出帧时间的替代品

原因：

1. `FixedUpdate` 更适合固定步长的逻辑推进
2. 在一次渲染帧之间，`FixedUpdate` 可能执行 `0` 次、`1` 次或多次
3. 视觉刺激最终仍然受显示器刷新率和窗口提交时机约束

建议职责划分：

`FixedUpdate` 负责：

1. `P300` block 计数
2. 行列刺激序列推进
3. flash / interval 状态切换
4. 暂停 / 恢复后的逻辑时钟维护

`Update` / 渲染负责：

1. 根据当前逻辑状态刷新刺激实体的可见性
2. 把当前刺激内容提交到播放窗口
3. 同步页面状态文本和调试信息

更稳的做法是把刺激时间量化到显示帧，而不是只按毫秒累计。

建议：

1. 根据目标显示器刷新率估算帧时长
2. 将 `flash_duration_ms` 和 `inter_stimulus_interval_ms` 转换为帧数
3. 运行时按“持续多少帧”推进刺激，而不是只比较裸毫秒
4. 日志中同时记录：
   1. 计划刺激开始时刻
   2. 实际切换到下一刺激的逻辑 tick
   3. 当前刺激对应的帧计数

例如：

1. `60 Hz` 显示器单帧约 `16.67 ms`
2. `100 ms` flash 可以近似量化为 `6` 帧
3. `75 ms` 间隔可以近似量化为 `4` 到 `5` 帧

首版实现建议：

1. 先采用 `FixedUpdate` 管理实验逻辑
2. 在参数层增加“按帧量化”的内部转换
3. 页面上仍显示毫秒参数，避免 UI 复杂化
4. 后续如果要继续提高精度，再评估：
   1. 显示器真实刷新率探测
   2. 帧提交时间记录
   3. 外部同步信号

## 8. 与现有模块的集成方式

- `Paradigm`

同时在菜单栏和 `i18n` 中增加对应入口。

2. 导出刺激播放时间线

## 9. 页面设计建议

建议首版先做一个统一范式测试页，而不是立即拆成多个业务页。

页面布局建议：

1. 左侧参数区
2. 中间 `gif` 自动显示区
3. 右侧状态与控制区

建议至少包含以下区域：

1. 范式参数区：当前首版主要是 `P300`
2. `gif` 自动显示区
3. 显示器选择区
4. 播放控制区：开始、暂停、恢复、退出
5. 状态区：当前会话状态、目标显示器、当前素材

对 `P300` 模式，参数区至少包含：

1. 目标字符

## 11. 风险与应对

### 11.1 多窗口与显示器适配

风险：

1. 不同显示器缩放比例可能不同
2. 目标显示器临时断开会导致播放目标失效

应对：

1. 播放前重新校验显示器列表
2. 全屏播放窗口使用独立逻辑尺寸和统一背景
3. 发现目标显示器失效时直接退出播放并提示

### 11.2 时序精度

风险：

1. `P300` 对刺激时序比较敏感
2. 主线程阻塞会影响闪烁稳定性
3. `FixedUpdate` 本身不等于显示器真实出帧时刻

应对：

1. 刺激推进统一交给独立播放状态机
2. 使用 `FixedUpdate` 管理逻辑，用渲染帧负责实际显示
3. 避免在播放循环中做阻塞文件读取
4. 素材解码和参数准备在播放前完成
5. 内部优先按帧量化刺激时长，而不是只按毫秒比较

### 11.3 范式扩展失控

风险：

1. 如果首版就支持过多范式，结构会很快膨胀

应对：

1. 首版只固定支持 `P300`，`gif` 仅作为页面素材显示
2. 统一会话与状态机
3. 后续新范式必须复用同一播放控制接口

## 12. 最终结论

范式模块建议独立为 `paradigm` crate，并以“主窗口 `gif` 素材显示 + 统一显示器播放底座 + `P300` 首范式”作为首版落地方向。

首版先把以下能力做稳：

1. 主窗口 `gif` 自动显示
2. 选定显示器全屏播放底座
3. 播放 / 暂停 / 恢复 / 退出
4. `P300` 最小闭环

这样后续无论要扩展更多视觉刺激范式，还是接入采集和实验记录，都有一个清晰、稳定的基础层可以继续演进。
