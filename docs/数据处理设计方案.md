# 数据处理设计方案

## 1. 概述

数据处理模块用于给当前项目提供一组可复用的基础信号处理能力，服务对象主要是：

1. 实时波形
2. 回放波形
3. 后续脑电/生理信号分析页面

根据 `docs/requirements.md`，首批目标包含：

1. 生成正弦波
2. FFT
3. IIR/FIR 设计
4. 滤波
5. 功率谱

首版目标不是做完整 DSP 平台，而是先建立一个稳定、清晰、可扩展的数据处理基础库，优先解决“算法可复用”和“UI 可调用”这两个问题。

## 2. 设计结论

建议新增独立 crate：

- `signal_processing`

这个 crate 只负责：

- 信号生成
- 频域分析
- 滤波器设计
- 滤波执行
- 结果数据结构

不负责：

- Bevy UI
- 文件读写
- 页面状态管理
- 长流程业务编排

这样做的原因很直接：

1. 算法和 UI 解耦
2. 方便在实时波形、回放波形、后续分析页之间复用
3. 方便单元测试
4. 后续如需引入更多 DSP 能力，边界清楚

## 3. 库选型建议

首版建议采用“轻量数据结构 + 成熟基础库”的路线。

建议组合：

- FFT：`rustfft`
- Example 画图：`plotters`
- 滤波器设计与执行：首版优先在 `signal_processing` 内自行封装

其中：

- `rustfft` 负责 FFT 计算本身
- `plotters` 负责 example 输出频谱图、时域曲线图
- 公共数据结构统一以 `Vec<f32>` 为主

### 3.1 FFT

推荐：

- `rustfft`

原因：

1. Rust 生态里使用广、成熟度高
2. 只解决 FFT 本身，职责单一
3. 和一维 `Vec<f32>` 数据流天然匹配

### 3.2 plotters

推荐：

- `plotters`

原因：

1. 适合在 `examples/` 中直接生成 PNG 或 SVG 图
2. 对频谱图、时域折线图这类静态图表支持直接
3. 不需要把画图能力耦合进 Bevy 主程序

建议用途限定为：

1. `examples/fft_demo.rs` 输出频谱图
2. `examples/filter_demo.rs` 输出滤波前后对比图
3. 人工验收算法结果

### 3.3 滤波器设计

首版建议：

- 先在项目内实现 FIR 窗函数法
- IIR 先支持常用二阶节或巴特沃斯低/高/带通的固定设计入口

原因：

1. 需求里写的是 “IIR/FIR 设计”，重点是给上层页面可调用的能力
2. 首版不需要把所有滤波器家族一次性做全
3. 把对外接口先定稳，比一开始追求算法覆盖面更重要

## 4. 整体架构

建议目录结构：

```text
signal_processing/
├── Cargo.toml
├── examples/
│   ├── generate_sine.rs
│   ├── fft_demo.rs
│   └── filter_demo.rs
└── src/
    ├── lib.rs
    ├── error.rs
    ├── types.rs
    ├── generator.rs
    ├── fft.rs
    ├── spectrum.rs
    ├── fir.rs
    ├── iir.rs
    └── filter.rs
```

职责划分：

| 文件 | 职责 |
|------|------|
| `types.rs` | 统一数据结构，如时域信号、频谱点、滤波器配置、滤波结果 |
| `generator.rs` | 生成正弦波等测试信号 |
| `fft.rs` | FFT 封装 |
| `spectrum.rs` | 功率谱、幅度谱等频谱后处理 |
| `fir.rs` | FIR 系数设计 |
| `iir.rs` | IIR 系数设计 |
| `filter.rs` | 对输入序列执行滤波 |
| `error.rs` | 统一错误类型 |
| `examples/` | 运行级示例，用于算法联调、人工验收和后续页面接入参考 |

## 5. 核心数据结构

```rust
pub struct SignalBuffer {
    pub sample_rate: f32,
    pub samples: Vec<f32>,
}

pub struct SineWaveConfig {
    pub sample_rate: f32,
    pub frequency: f32,
    pub amplitude: f32,
    pub phase: f32,
    pub duration_secs: f32,
}

pub struct FftOutput {
    pub sample_rate: f32,
    pub bins: Vec<Complex32>,
}

pub struct SpectrumPoint {
    pub frequency: f32,
    pub value: f32,
}

pub enum FilterKind {
    LowPass,
    HighPass,
    BandPass,
    BandStop,
}

pub struct FirDesignConfig {
    pub sample_rate: f32,
    pub filter_kind: FilterKind,
    pub cutoff_hz: Vec<f32>,
    pub tap_count: usize,
}

pub struct IirDesignConfig {
    pub sample_rate: f32,
    pub filter_kind: FilterKind,
    pub cutoff_hz: Vec<f32>,
    pub order: usize,
}
```

说明：

1. 首版统一使用 `f32`
2. 首版统一按单通道一维序列处理
3. 公共接口先保持轻量
4. 如果未来确实进入多通道矩阵计算阶段，再单独升级数据结构层

## 6. 各能力设计

### 6.1 生成正弦波

提供一个纯函数接口：

```rust
pub fn generate_sine_wave(config: &SineWaveConfig) -> Result<SignalBuffer, SignalProcessError>
```

输出直接是：

- 采样率
- 样本数组

用途：

1. 实时波形页测试输入
2. 回放波形页测试输入
3. FFT / 滤波 / 功率谱的算法联调输入

### 6.2 FFT

建议接口：

```rust
pub fn compute_fft(signal: &SignalBuffer) -> Result<FftOutput, SignalProcessError>
```

首版约束：

1. 输入为 `Vec<f32>`
2. FFT 直接构造 `Vec<Complex32>` 工作缓冲区
3. 输出保留为 `Vec<Complex32>`
4. 后续如需要窗函数和补零，再在上层配置中扩展

### 6.3 功率谱

建议在 FFT 结果之上提供：

```rust
pub fn compute_power_spectrum(
    signal: &SignalBuffer,
) -> Result<Vec<SpectrumPoint>, SignalProcessError>
```

首版先提供：

1. 单边功率谱
2. 频率轴换算
3. 基础幅值归一化

不在首版里做：

1. Welch
2. 多段平均
3. 复杂窗函数组合

### 6.4 FIR 设计

首版建议优先支持：

1. 低通
2. 高通
3. 带通

方法：

- 窗函数法

建议接口：

```rust
pub fn design_fir(config: &FirDesignConfig) -> Result<Vec<f32>, SignalProcessError>
```

原因：

1. 实现难度可控
2. 对 EEG/生理信号类桌面工具已经足够实用
3. 设计结果就是系数数组，方便后续滤波执行复用

### 6.5 IIR 设计

首版建议优先支持：

1. 巴特沃斯低通
2. 巴特沃斯高通
3. 巴特沃斯带通

建议接口：

```rust
pub fn design_iir(config: &IirDesignConfig) -> Result<IirCoefficients, SignalProcessError>
```

首版不建议上太多滤波器族，否则实现和验证成本会快速膨胀。

### 6.6 滤波

建议统一提供：

```rust
pub fn apply_fir(signal: &SignalBuffer, coeffs: &[f32]) -> Result<SignalBuffer, SignalProcessError>

pub fn apply_iir(
    signal: &SignalBuffer,
    coeffs: &IirCoefficients,
) -> Result<SignalBuffer, SignalProcessError>
```

首版建议只做离线序列滤波，不急着做：

1. 实时流式滤波状态缓存
2. 多线程块处理
3. GPU 加速

## 7. 与现有页面的集成方式

### 7.1 实时波形

后续可以增加：

1. 用正弦波替代随机数，作为更稳定的测试输入
2. 对实时数据做简单频谱预览

### 7.2 回放波形

后续可以增加：

1. 打开 EDF/BDF 后对当前页做 FFT
2. 对当前通道做 FIR/IIR 滤波
3. 叠加显示滤波前后结果
4. 生成功率谱视图

### 7.3 医学影像

当前不直接耦合，数据处理模块首版主要服务波形/时序信号，不和体数据处理混在一起。

## 8. 数据结构原则

当前阶段的数据处理能力仍然以一维时序信号为主，因此原则改为：

1. 公共输入输出优先使用 `Vec<f32>`
2. `rustfft` 的 `Complex32` 只在 FFT 结果和相关频域处理中使用
3. 不为了未来可能的矩阵计算，提前把首版接口做重

这样做的好处是：

1. 首版实现更轻
2. 跟 `rustfft` 对接更直接
3. 示例代码和单元测试都更容易写

## 9. UI 落地建议

首版不建议先单独做一个“大而全的数据处理页面”，这次先不接 UI，先通过 example 完成验证：

### Phase 1

- 先完成纯算法 crate
- 用单元测试验证结果

### Phase 2

- 用 `examples/` 验证正弦波生成、FFT 和滤波结果

### Phase 3

- 再逐步扩展到更多滤波器类型和频谱能力

这样可以避免先做 UI，再发现底层接口和算法结果还没稳定。

## 10. Example 说明

`signal_processing/examples/` 不只是演示 API，还承担三类职责：

1. 人工验收算法结果
2. 给后续 UI 接入提供最小调用样例
3. 给调试和回归提供固定输入

首版建议使用固定、容易验证的合成信号，而不是一开始就依赖真实 EDF/BDF 文件。

### 10.1 generate_sine.rs

用途：

- 验证正弦波生成功能

建议信号：

- 单频正弦波
- 采样率：`256 Hz`
- 频率：`10 Hz`
- 幅值：`1.0`
- 时长：`2 s`

原因：

1. 参数简单
2. 容易人工判断是否正确
3. 后续可以直接复用到 FFT 示例

### 10.2 fft_demo.rs

用途：

- 验证 FFT 和功率谱输出
- 输出频谱图

建议信号：

- 双频正弦叠加
- 采样率：`256 Hz`
- 时长：`4 s`
- 分量 1：`10 Hz`，幅值 `1.0`
- 分量 2：`40 Hz`，幅值 `0.4`

期望现象：

1. 频谱图在 `10 Hz` 附近出现主峰
2. 在 `40 Hz` 附近出现次峰
3. 其余频段接近噪声底

这样可以快速验证：

1. FFT 频率轴是否正确
2. 幅值关系是否大体合理
3. `plotters` 输出图是否可读

### 10.3 filter_demo.rs

用途：

- 验证 FIR/IIR 设计与滤波执行
- 输出滤波前后时域图或频谱图

建议信号：

- `10 Hz` 有用信号 + `50 Hz` 干扰信号
- 采样率：`256 Hz`
- 时长：`4 s`
- `10 Hz` 幅值：`1.0`
- `50 Hz` 幅值：`0.5`

建议示例滤波器：

- 低通截止频率：`20 Hz`

期望现象：

1. 滤波后 `10 Hz` 主成分基本保留
2. `50 Hz` 干扰明显减弱
3. 频谱图中高频峰值下降明显

### 10.4 后续可增加的 example

后续如果数据处理能力继续扩展，可以再补：

1. `power_spectrum_demo.rs`
2. `bandpass_demo.rs`
3. `multi_channel_mock_demo.rs`

但首版不建议一次做太多 example，优先把：

1. 生成正弦波
2. FFT 频谱图
3. 滤波前后对比

这三条跑通。

## 11. 实现步骤

### Phase 1: 基础库骨架

1. 新增 `signal_processing` crate 并加入 workspace
2. 在 `Cargo.toml` 中加入 `rustfft`、`plotters`
3. 新增 `lib.rs`、`error.rs`、`types.rs`
4. 定义统一错误类型和基于 `Vec<f32>` 的输入输出数据结构
5. 新增 `examples/generate_sine.rs`
6. 补基础单元测试骨架

### Phase 2: 生成与频谱

1. 实现正弦波生成
2. 接入 `rustfft`
3. 实现 `Vec<f32> -> Vec<Complex32>` 的 FFT 封装
4. 实现单边功率谱输出
5. 新增 `examples/fft_demo.rs`，并用 `plotters` 输出频谱图
6. 补频率轴和幅值正确性的单元测试

### Phase 3: FIR/IIR 设计

1. 实现 FIR 窗函数法设计
2. 实现常用 IIR 设计入口
3. 定义滤波器配置结构
4. 新增 `examples/filter_demo.rs`，并用 `plotters` 输出滤波前后对比图
5. 补设计结果稳定性测试

### Phase 4: 滤波执行

1. 实现 FIR 序列滤波
2. 实现 IIR 序列滤波
3. 补输入边界和异常参数测试
4. 补与测试信号联调用例

## 12. 风险与应对

### 10.1 算法覆盖面过大

风险：

- 如果一开始就想支持太多滤波器和谱分析变种，工程量会迅速失控

应对：

- 首版只覆盖需求里最核心的几类能力
- 先把接口做稳，再扩展算法族

### 12.2 UI 和算法耦合过深

风险：

- 如果直接把处理逻辑写进页面系统里，后续复用会很差

应对：

- 算法统一收敛到独立 crate
- UI 只做参数输入和结果展示

### 12.3 结果正确性难验证

风险：

- 信号处理最怕“能跑但不对”

应对：

- 优先补单元测试
- 用正弦波、已知频率样本和固定滤波器参数做基准验证

## 13. 最终结论

数据处理模块建议落成一个独立的 `signal_processing` crate，首版先聚焦：

1. 正弦波生成
2. FFT
3. 单边功率谱
4. 常用 FIR/IIR 设计
5. 离线序列滤波

实现路线建议是：

1. `signal_processing` 作为纯算法层
2. 公共接口以 `Vec<f32>` 为主
3. `rustfft` 负责 FFT 底层实现
4. `plotters` 负责 `examples/` 的频谱图和对比图输出
5. 先完成算法、测试和 examples 验证
