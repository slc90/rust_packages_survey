# 音频播放设计方案

## 1. 概述

音频播放功能用于在当前 `Bevy 0.18 + ECS + Plugin` 桌面应用中加载和播放本地音频文件。根据当前 `docs/requirements.md`，首版先聚焦两项能力：

1. 读取本地 `mp3`
2. 播放 / 暂停

首版目标不是做完整音乐播放器，而是先完成一个稳定、可扩展的最小闭环：

1. 能从文件弹窗选择本地音频文件
2. 能正常播放和暂停
3. 能显示基础状态信息，例如文件名、播放状态、时长和当前位置
4. 能为后续扩展进度条、音量、频谱分析保留清晰边界

## 2. 库选型结论

当前实现固定采用：

- 播放库：`rodio`

依赖关系上，`rodio` 底层已经建立在音频输出库和解码能力之上：

- 输出能力来自 `cpal`
- 默认解码能力可接入 `Symphonia`

首版不建议直接自己拼：

- `cpal + symphonia`

也不建议首版就上更重的游戏音频方案：

- `kira`

## 3. 为什么评审阶段优先 rodio

### 3.1 和当前需求最匹配

当前需求只有：

- 打开本地 `mp3`
- 播放
- 暂停

这类需求更接近“桌面播放器最小闭环”，`rodio` 已经提供了足够直接的播放抽象，工程量明显小于自己维护底层输出流和解码循环。

### 3.2 工程复杂度最低

如果直接用 `cpal`，需要自己处理：

- 输出设备枚举
- 输出流格式协商
- 回调线程里的样本填充
- 播放状态同步
- 缓冲和欠载问题

如果直接用 `symphonia + cpal`，还要再加上：

- 容器探测
- 解码循环
- PCM 重采样或格式适配

对于当前项目阶段，这些都不是首版必须承担的复杂度。

### 3.3 后续仍然可扩展

首版采用 `rodio` 不会堵死后路。后续如果要加：

- 频谱分析
- 音量调节
- 播放进度拖动
- 多音频并发

可以先在当前封装层继续扩展；如果未来确实需要更细粒度控制，再把内核替换成 `symphonia + cpal` 也比较清晰。

## 4. 参考库情况

### 4.1 rodio

适合作为首版播放内核。官方文档强调它是音频播放库，使用方式就是创建输出流、解码音频源并交给播放器/混音器播放。  
来源：

- `rodio` docs.rs: https://docs.rs/rodio/latest/rodio/
- `rodio` GitHub: https://github.com/RustAudio/rodio

### 4.2 symphonia

适合作为解码层或后续增强路线。官方文档说明它是纯 Rust 的媒体容器和音频解码框架，适合在需要更细粒度解码控制时使用。  
来源：

- `symphonia` docs.rs: https://docs.rs/symphonia/latest/symphonia/

### 4.3 cpal

适合作为底层音频输入输出能力，不适合作为当前需求的首版直接入口。官方文档说明它是纯 Rust 的跨平台音频 I/O 库。  
来源：

- `cpal` docs.rs: https://docs.rs/cpal/latest/cpal/

### 4.4 kira

更偏游戏音频系统，适合复杂混音、时钟和空间音频，不是当前“播放 mp3”最省事的方案。  
来源：

- `kira` docs.rs: https://docs.rs/kira/latest/kira/

## 5. 整体架构

建议新增一个内部 crate，专门负责音频播放内核：

```text
audio_player/
├── Cargo.toml
└── src/
    ├── lib.rs
    ├── player.rs
    ├── events.rs
    ├── state.rs
    └── error.rs

ui/src/homepage/audio_player/
├── components.rs
├── resources.rs
├── systems.rs
└── plugin.rs
```

分层职责：

| 层级 | 职责 |
|------|------|
| `audio_player` crate | 文件加载、播放句柄、播放状态、后台播放线程 |
| `ui::homepage::audio_player` | 页面生命周期、文件选择、按钮交互、状态显示 |
| Bevy 主线程 | 页面渲染、状态同步、用户输入 |

## 6. 核心数据结构

```rust
#[derive(Resource, Default)]
pub struct AudioPlaybackState {
    pub current_player: Option<AudioPlayerEntry>,
    pub initial_directory: std::path::PathBuf,
}

pub struct AudioPlayerEntry {
    pub source: std::path::PathBuf,
    pub status: AudioPlaybackStatus,
    pub duration_ms: Option<u64>,
    pub position_ms: u64,
    pub player: Option<audio_player::PlayerHandle>,
}

pub enum AudioPlaybackStatus {
    Idle,
    Loading,
    Playing,
    Paused,
    Ended,
    Error,
}
```

说明：

- 首版只维护一个音频播放实例
- 不做多音频并发
- 不做播放列表

## 7. 与 Bevy 的集成方式

### 7.1 线程边界

建议拆成两部分：

1. 音频播放后台线程
2. Bevy 主线程

后台线程负责：

- 初始化 `rodio` 输出流
- 加载并解码音频源
- 执行播放 / 暂停命令
- 上报结束和错误事件

Bevy 主线程负责：

- 文件弹窗
- 页面 UI
- 状态同步
- 文本显示

### 7.2 状态同步策略

建议通过线程安全通道同步：

- UI -> 播放线程：控制命令
- 播放线程 -> UI：状态事件

首版至少需要这些命令：

- `Load(PathBuf)`
- `Play`
- `Pause`
- `Close`

首版至少需要这些事件：

- `Loaded`
- `StatusChanged`
- `PositionUpdated`
- `Ended`
- `Error`

```text
+------------------------------------------------------------------+
|  [主页] 播放音频                                                   |
+------------------------------------------------------------------+
| [打开文件] [播放/暂停] [关闭音频]                                  |
+------------------------------------------------------------------+
|                                                                  |
|  文件: sample.mp3                                                |
|  状态: 播放中                                                     |
|  位置: 00:01:18 / 00:03:42                                       |
|                                                                  |
|  预留区域:                                                        |
|  - 后续可加入进度条                                               |
|  - 后续可加入音量                                                 |
|  - 后续可加入频谱或波形预览                                       |

```

交互约束：

- 页面只维护一个当前音频文件
- 点击“打开文件”后通过原生文件弹窗选择 `mp3`
- 点击“播放/暂停”只切换当前文件状态
- 点击“关闭音频”后释放当前播放资源

## 10. 风险与应对

### 10.1 时长和位置精度不够细

风险：

- `rodio` 首版更适合快速播放闭环，不适合一开始就做高精度进度控制

应对：

- 首版先保证播放/暂停稳定
- 拖动进度条放到后续阶段
- 如果后续对精准位置要求更高，再评估切到 `symphonia + cpal`

### 10.2 不同平台音频设备差异

风险：

- 默认输出设备在不同系统上行为可能不完全一致

应对：

- 首版先承诺桌面平台开发环境
- 统一从默认输出设备开始
- 后续再评估是否支持设备切换

## 11. 最终结论

当前仓库实际实现采用：

- 播放内核：`rodio`
- 音频输出：`cpal`
- 解码能力：`Symphonia`
- UI 与交互：`Bevy 0.18`

本次实际落地路线是：

- 使用文件弹窗选择本地音频
- 支持播放 / 暂停
- 单实例维护当前音频文件和播放状态
- 保留后续扩展进度条、音量和频谱的 UI 边界
