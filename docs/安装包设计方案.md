# 安装包设计方案

## 1. 概述

安装包模块用于把当前项目从“开发环境里可运行的 Rust 工程”整理成“用户可直接安装和启动的桌面应用”。

根据 `docs/requirements.md`，当前目标只有一项：

1. 安装包

但这项能力实际包含的内容不少，至少要覆盖：

1. 程序主可执行文件
2. 配置文件
3. 运行时依赖
4. 必需模型文件
5. 卸载入口
6. 安装目录规范

首版目标不是做企业级安装器平台，而是先完成一个稳定、可维护的 Windows 安装闭环：

1. 能生成可安装的 `Windows` 安装包
2. 安装后可直接启动程序
3. 嵌入资源、运行时依赖和必需模型可随安装包一起分发
4. 配置文件和输出目录有明确落点

## 2. 设计结论

当前项目首版安装包建议固定采用：

- `WiX Toolset v5`

也就是基于 `WiX Toolset v5` 生成标准 `msi` 安装包。

不再把首版方案分散到多个安装器路线中。

原因：

1. 当前项目主目标平台明显是 `Windows`
2. `msi` 更符合桌面程序安装、卸载、开始菜单快捷方式等需求
3. `WiX v5` 仍然是官方主线，避免继续依赖已经过时的 `cargo-wix` / `WiX v3` 工具链
4. 后续如果要接 CI 发布，`msi` 也更容易标准化

同时明确一条发布约束：

1. 安装包首版必须做到“用户安装后直接可用”
2. 不要求用户额外下载 `GStreamer Runtime`
3. 不依赖安装时联网下载第三方运行时

## 3. 目标与边界

首版安装包必须解决的问题：

1. 安装主程序
2. 放置默认配置
3. 放置必要运行时资源
4. 创建开始菜单和桌面快捷方式
5. 支持卸载

首版不做：

1. 自动在线更新
2. 多语言安装向导
3. 驱动级依赖安装
4. 多平台统一安装方案

## 4. 为什么优先选 `WiX Toolset v5`

### 4.1 和当前项目结构更匹配

当前仓库已经是完整 workspace，真正需要交付给用户的是：

1. `entry` 生成的主程序
2. `config/config_file/config.json`
3. `deepl_models/whisper_base/` 下的必需模型文件
4. 如有必要的外部运行时文件

`WiX v5` 更适合把这些内容按安装目录组织起来，而不是只打一个压缩包。

### 4.2 安装器语义更完整

相比“打 zip 让用户自己解压”，`msi` 更直接支持：

1. 安装路径选择
2. 程序组入口
3. 卸载注册
4. 版本升级

### 4.3 对当前项目更稳妥

当前程序已经包含：

1. `Bevy`
2. `GStreamer` 相关视频播放能力
3. 嵌入式资源
4. `Whisper` 本地推理模型
5. 本地配置文件

这些内容都更适合通过安装器显式管理，而不是依赖用户手工复制目录。

## 5. 安装包内容设计

建议首版安装目录结构如下：

```text
安装目录/
├── rust_packages_survey.exe
├── config_file/
│   └── config.json
├── deepl_models/
│   └── whisper_base/
│       ├── config.json
│       ├── model.safetensors
│       ├── tokenizer.json
│       ├── vocab.json
│       └── ...
├── logs/
├── screenshots/
├── deep_learning_output/
└── report_output/
```

说明：

1. 主程序放在安装目录根下
2. 当前配置读取逻辑已经基于 `exe` 同级目录寻找 `config_file/config.json`
3. `whisper_base` 模型目录需要和当前深度学习模块的查找逻辑保持一致
4. 日志、截图、报告、深度学习输出目录建议安装后自动创建或首次运行自动创建

## 6. 资源与运行时策略

### 6.1 嵌入资源

当前项目已经有 `embedded_assets` crate，图标、字体、shader、范式 `gif` 等内容已经可以直接嵌入到程序中。

因此安装包首版不需要再额外复制这些静态资源目录。

结论：

1. `embedded_assets` 中的资源随可执行文件一起分发
2. 安装包不再单独携带这类静态资源目录

### 6.2 配置文件

配置文件仍然建议保留为外部文件：

1. 默认配置通过安装包写入 `config_file/config.json`
2. 用户修改配置后，不应要求重新打包程序
3. 卸载时是否删除用户配置，建议作为安装器选项或保守保留

### 6.3 第三方运行时

当前最大的安装风险来自视频播放依赖。

仓库已有 `media_player`，而视频播放依赖 `GStreamer`。

首版不再保留“让用户自行安装运行时”的路线，直接固定为：

1. 安装包内捆绑 `GStreamer Runtime`
2. 用户安装应用时不需要额外下载 `GStreamer`
3. 运行时文件由安装包构建流程提前整理完成

### 6.4 深度学习模型

当前程序里的 `Whisper` 推理不是纯代码能力，还依赖本地模型目录。

已实现链路明确使用：

1. `deepl_models/whisper_base/`

因此安装包首版必须把 `whisper_base` 一并分发，否则程序安装后深度学习模块无法正常运行。

结论：

1. `whisper_base` 作为必需运行资源进入安装包
2. 安装后目录结构必须保持和当前程序查找逻辑一致
3. 不要求用户首次启动后再手工下载模型

### 6.5 CUDA 运行时

当前 `Whisper` 推理链路已经固定为 `CUDA` 路线，因此安装包不能只带模型，不带 `CUDA` 运行时库。

这里要明确区分两类依赖：

1. 可随安装包分发的 `CUDA` 运行时 `dll`
2. 必须由用户机器自行满足的 `NVIDIA` 显卡驱动

首版方案结论：

1. 安装包内捆绑程序实际依赖的 `CUDA` 运行时 `dll`
2. 不尝试在安装包里分发或替换 `NVIDIA` 驱动
3. 程序启动或进入深度学习页面时，要对 `CUDA` 环境可用性做显式检测

这样才能同时满足“装完即用”和“失败时能解释清楚”的要求。

## 7. `GStreamer` 安装策略

如果继续保留当前视频播放能力，安装包必须处理 `GStreamer` 运行时问题。

建议首版方案：

1. 固定一个官方 `Windows msvc x86_64` 版 `GStreamer Runtime`
2. 从官方安装包或官方 `merge-modules.zip` 中整理所需文件
3. 把运行时所需的 `dll` 和插件目录并入 `msi`
4. 安装后直接从程序目录加载运行时，不要求用户单独配置环境

不采用以下路线：

1. 让用户手工安装 `GStreamer Runtime`
2. 安装时再在线下载 `GStreamer`
3. 首版先只做“主程序安装，视频功能另配环境”

因为这些路线都会明显削弱“装完即用”的目标。

### 7.1 运行时来源

`GStreamer` 运行时文件统一从官方 Windows 发布包取得，不从开发机系统目录临时搜集。

推荐来源：

1. 官方 `runtime installer` 安装后的目录
2. 官方 `merge-modules.zip`

首版更推荐第二种，因为更适合做可重复的离线整理脚本。

整理来源目录时，重点关注：

1. `bin/` 下的核心运行时 `dll`
2. `lib/gstreamer-1.0/` 下的插件 `dll`

如果某些插件还依赖额外第三方 `dll`，也要一并纳入 staging 目录。

### 7.2 首版目录策略

建议把 `GStreamer` 运行时整理到安装目录中的专用子目录，例如：

```text
安装目录/
├── rust_packages_survey.exe
├── config_file/
│   └── config.json
├── gstreamer/
│   ├── bin/
│   └── lib/
└── ...
```

这样做的目的：

1. 运行时文件和主程序一起分发
2. 目录来源清晰，便于后续升级固定版本
3. 便于构建脚本做增量整理和校验

程序启动时需要保证能定位到这套运行时目录。实现方式可以是：

1. 在启动脚本或安装器中补充运行时搜索路径
2. 在程序启动阶段显式设置 `GStreamer` 所需环境变量

首版优先选实现成本更低、行为更稳定的一种，不要求用户自己改系统环境变量。

## 8. 安装器行为设计

首版安装器建议支持以下行为：

1. 选择安装目录
2. 安装主程序
3. 写入默认配置文件
4. 安装 `whisper_base` 模型文件
5. 安装程序依赖的 `CUDA` 运行时 `dll`
6. 创建开始菜单快捷方式
7. 可选创建桌面快捷方式
8. 注册卸载信息

卸载行为建议：

1. 删除程序文件
2. 删除快捷方式
3. 保守做法是不主动删除用户生成的数据目录

保留的数据目录建议包括：

1. `logs/`
2. `screenshots/`
3. `deep_learning_output/`
4. 用户修改过的 `config_file/config.json`

安装目录中的 `deepl_models/whisper_base/` 不属于用户输出目录，默认应视为程序运行资源，由安装器统一安装和卸载。

`CUDA` 运行时 `dll` 也属于程序运行资源，应和主程序一起安装和卸载，不要求用户手工维护。

## 9. 版本与升级策略

首版就应该明确版本号和升级逻辑。

建议：

1. 安装包版本直接复用 workspace 版本
2. 每次发布时生成带版本号的 `msi`
3. 同一产品升级时允许覆盖安装

这样后续更容易接 GitHub Release 或 CI 自动发布。

`CUDA` 运行时和 `whisper_base` 模型也应和应用版本一起做兼容性绑定，避免应用升级后运行时版本漂移。

## 10. CUDA 运行时策略

### 10.1 打包范围

安装包首版需要把程序实际依赖的 `CUDA` 运行时 `dll` 一并带上。

具体名单不要靠人工拍脑袋固定，而应通过当前 `Whisper` 推理链路的实际依赖来整理 staging 目录。

常见会涉及的 `CUDA` 运行时文件通常包括：

1. `cudart` 相关 `dll`
2. `cublas` / `cublasLt` 相关 `dll`
3. 其他被当前推理链路动态依赖的 `CUDA` 库

首版要求：

1. 只打包程序实际需要的运行时文件
2. 文件来源固定为已验证版本的 `CUDA Toolkit`
3. 由脚本整理到 staging 目录，再纳入 `msi`

### 10.2 不纳入安装包的部分

以下内容不由安装包负责分发：

1. `NVIDIA` 显卡驱动
2. 不被当前程序实际使用的整套 `CUDA Toolkit`
3. 开发工具链，例如编译器、头文件和样例工程

原因：

1. 驱动属于系统级依赖，不适合由应用安装包替代
2. 整套 `CUDA Toolkit` 体积过大，也超出运行时分发所需范围

### 10.3 运行前检测

即使安装包带上了 `CUDA` 运行时 `dll`，程序也仍然需要检测用户机器是否满足推理前置条件。

至少要覆盖：

1. 是否存在可用的 `NVIDIA GPU`
2. 驱动是否满足当前 `CUDA` 版本要求
3. 程序依赖的 `CUDA` 库是否成功加载

如果检测失败，程序需要给出明确提示，例如：

1. 模型和程序已安装完成
2. 当前机器缺少可用 `CUDA` 环境
3. 深度学习推理功能不可用，但其他功能可继续使用

## 11. 仓库落地建议

建议新增安装包相关目录：

```text
installer/
├── wix/
│   ├── main.wxs
│   └── License.rtf
└── scripts/
    ├── build_installer.ps1
    └── prepare_runtime.ps1
```

职责建议：

| 文件 | 职责 |
|------|------|
| `installer/wix/main.wxs` | `WiX v5` 安装器定义 |
| `installer/wix/License.rtf` | 安装器许可说明 |
| `installer/scripts/build_installer.ps1` | 一键构建安装包 |
| `installer/scripts/prepare_runtime.ps1` | 从官方 `GStreamer` 包整理运行时文件并生成 staging 目录 |
| `installer/scripts/prepare_models.ps1` | 整理 `deepl_models/whisper_base/` 并生成模型 staging 目录 |
| `installer/scripts/prepare_cuda_runtime.ps1` | 整理程序实际依赖的 `CUDA` 运行时 `dll` 并生成 staging 目录 |

## 12. 实现步骤

### Phase 1: 跑通最小安装包

1. 安装 `WiX Toolset v5` 的 `wix` .NET tool
2. 缓存 `WixToolset.UI.wixext`
3. 生成基础 `wxs` 模板
5. 把 `entry` 的 release 可执行文件纳入安装内容
6. 把 `config/config_file/config.json` 纳入安装内容
7. 把 `deepl_models/whisper_base/` 纳入安装内容
8. 把程序依赖的 `CUDA` 运行时 `dll` 纳入安装内容
9. 生成最小 `msi`
10. 验证安装、启动、卸载流程

### Phase 2: 补安装目录与快捷方式

1. 配置默认安装目录
2. 增加开始菜单快捷方式
3. 增加桌面快捷方式选项
4. 调整安装器产品名、图标和版本号
5. 验证从开始菜单启动是否正常

### Phase 3: 处理运行时依赖

1. 固定一个官方 `GStreamer Windows msvc x86_64` 版本
2. 从官方 `runtime installer` 或 `merge-modules.zip` 提取运行时文件
3. 梳理视频播放实际需要的 `dll` 和插件目录
4. 用 `prepare_runtime.ps1` 生成安装包 staging 目录
5. 把 staging 中的运行时依赖并入安装包
6. 验证视频播放功能在纯净机器上是否可用

### Phase 4: 工程化

1. 新增安装包构建脚本
2. 新增 `prepare_models.ps1` 和 `prepare_cuda_runtime.ps1`
3. 把 release 构建和安装包构建串起来
4. 把运行时整理、模型整理和 `CUDA` 运行时整理脚本串进安装包构建流程
5. 增加安装后 `CUDA` 环境检查验收项
6. 更新 `structure.md`
7. 更新 `docs/requirements.md`

## 13. 风险与应对

### 12.1 `GStreamer` 运行时复杂

风险：

1. `dll` 和插件目录不完整时，视频播放会直接失效
2. 不同机器上依赖漂移时，手工拷贝目录很容易遗漏文件

应对：

1. 首版先固定一个已验证的 `GStreamer Runtime` 版本
2. 运行时文件只从官方包整理，不从开发机现有环境临时拼装
3. 安装包构建脚本中显式整理依赖文件

### 13.2 配置路径和输出目录耦合

风险：

1. 当前程序部分路径策略依赖 `exe` 同级目录

应对：

1. 安装包目录结构要和程序读取逻辑保持一致
2. 安装后首次运行要验证配置读取和日志输出路径

### 13.3 模型目录缺失会导致功能不可用

风险：

1. 如果漏掉 `deepl_models/whisper_base/`，安装后 `Whisper` 直接不可用
2. 模型目录名或层级变化时，当前程序可能找不到模型

应对：

1. 把 `whisper_base` 固定为安装包必选内容
2. 用独立脚本整理模型目录，避免手工复制遗漏
3. 安装后把深度学习页面纳入基础验收

### 13.4 CUDA 环境不完整会导致推理不可用

风险：

1. 即使 `dll` 已随包分发，用户机器没有可用 `NVIDIA` 驱动时仍然无法推理
2. `CUDA` 运行时版本和驱动版本不匹配时会启动失败

应对：

1. 安装包中只负责分发可再分发的 `CUDA` 运行时 `dll`
2. 程序启动阶段保留明确的 `CUDA` 可用性检测
3. 对失败原因给出清晰提示，不把问题伪装成模型缺失或程序崩溃

### 13.5 安装包越来越重

风险：

1. 如果把全部运行时和模型都打进去，安装包体积会明显增大

应对：

1. 首版先接受体积增加，优先保证“装完能跑”
2. 二期再评估按功能拆分或精简运行时内容

## 14. 最终结论

安装包首版建议固定采用 `WiX Toolset v5` 生成 `Windows MSI`。

首版最重要的是把下面几件事做稳：

1. 安装主程序
2. 放置默认配置文件
3. 把固定版本的 `GStreamer Runtime` 一并打入安装包
4. 把 `deepl_models/whisper_base/` 一并打入安装包
5. 把程序依赖的 `CUDA` 运行时 `dll` 一并打入安装包
6. 创建快捷方式和卸载入口

并且首版明确坚持：

1. 不要求用户单独下载 `GStreamer`
2. 不依赖安装时联网拉取运行时
3. 不要求用户手工补齐 `whisper_base` 模型
4. `CUDA` 驱动仍由用户机器满足，安装包只负责可再分发运行时
5. 以“装完即用”为安装包验收标准

这样才能把当前项目从“开发工程”真正推进到“可交付桌面应用”。
