# 截图设计方案

## 1. 概述

截图功能用于在当前 `Bevy 0.18 + ECS + Plugin` 桌面应用中导出可复用的图像结果，覆盖以下能力：

1. 程序窗口截图
2. 程序内区域截图
3. 程序所在桌面截图
4. 鼠标选定区域截图

首版目标不是一次性做成完整的系统截图工具，而是先完成一套稳定、可扩展的最小闭环：

1. 能稳定导出当前程序窗口 PNG
2. 能对程序内指定区域进行截图
3. 能为后续报告导出、问题定位、图像对比提供统一图片输出
4. 为后续扩展程序所在桌面截图和桌面框选截图保留清晰边界

## 2. 设计结论

截图功能建议拆成两条技术路线实现：

1. 程序内截图
2. 系统级截图

这样拆分的原因是：

1. `程序窗口截图` 和 `程序内区域截图` 的本质是导出应用自身渲染结果
2. `程序所在桌面截图` 和 `鼠标选定区域截图` 的本质是调用操作系统桌面抓屏能力
3. 两类能力的底层依赖、坐标体系、调试方式都不同，混在一起实现会放大复杂度

推荐交付顺序如下：

1. 程序窗口截图
2. 程序内区域截图
3. 程序所在桌面截图
4. 鼠标选定区域截图

这个顺序可以先交付与当前 Bevy 架构最贴近、风险最低的能力，再逐步增加系统级交互。

## 3. 能力范围定义

### 3.1 程序窗口截图

目标：

- 导出当前程序窗口当前帧内容为 PNG

特点：

- 截图结果只包含当前应用窗口内容
- 不依赖系统桌面抓屏
- 适合 UI、波形、医学影像三视图、Bevy 内视频纹理内容导出

限制：

- 通常不包含系统文件对话框、系统菜单等窗口外内容
- 若存在独立原生窗口，需要明确截图目标窗口

### 3.2 程序内区域截图

目标：

- 在当前程序窗口内对预定义的固定矩形区域导出 PNG

特点：

- 复用程序窗口截图结果
- 可以作为医学影像局部截图、波形局部截图、图表局部截图的通用能力

首版建议：

1. 只支持固定功能区导出
2. 固定区域由业务模块预先定义，不提供用户临时拖拽框选

### 3.3 程序所在桌面截图

目标：

- 导出当前程序所在显示器桌面的当前画面为 PNG

特点：

- 依赖 Windows 系统抓屏能力
- 可覆盖程序窗口外内容
- 只截取当前程序所在的显示器，不截取所有显示器拼接结果

适用场景：

- 记录当前程序所在桌面的操作环境
- 保存跨窗口协同画面
- 为问题反馈提供完整现场信息

边界约束：

1. 只按当前程序主窗口所在显示器抓图
2. 不处理程序窗口横跨多个显示器的情况
3. 如果程序位于多显示器环境中，默认以程序当前所在显示器为准

### 3.4 鼠标选定区域截图

目标：

- 用户在桌面上拖拽选区后导出该区域 PNG

特点：

- 本质上是“先抓整屏，再裁剪选区”
- 需要额外的遮罩层与交互层支持

这是四项能力里实现复杂度最高的一项，建议放在最后完成。

## 4. 整体架构

建议新增独立 crate：

- `screenshot`

建议结构如下：

```text
screenshot/
├── Cargo.toml
└── src/
    ├── lib.rs
    ├── request.rs
    ├── result.rs
    ├── error.rs
    ├── bevy_capture.rs
    ├── win_capture.rs
    ├── crop.rs
    └── save.rs

ui/src/homepage/screenshot/
├── components.rs
├── constants.rs
├── resources.rs
├── systems.rs
└── plugin.rs
```

分层职责：

| 层级 | 职责 |
|------|------|
| `screenshot` crate | 截图请求模型、抓图后端、图片裁剪、PNG 保存 |
| `ui::homepage::screenshot` | 页面交互、截图模式切换、固定区域选择、测试页面、保存反馈 |
| Windows 系统层 | 当前显示器抓图、桌面区域抓图、屏幕坐标查询 |

## 5. 核心数据结构建议

```rust
#[derive(Debug, Clone)]
pub enum ScreenshotMode {
    Window,
    RegionInWindow,
    CurrentDisplay,
    RegionOnScreen,
}

#[derive(Debug, Clone)]
pub struct ScreenshotRequest {
    pub mode: ScreenshotMode,
    pub target_window: Option<bevy::window::WindowRef>,
    pub region: Option<CaptureRegion>,
    pub file_name: String,
}

#[derive(Debug, Clone, Copy)]
pub struct CaptureRegion {
    pub x: u32,
    pub y: u32,
    pub width: u32,
    pub height: u32,
}

#[derive(Debug)]
pub struct ScreenshotResult {
    pub width: u32,
    pub height: u32,
    pub path: std::path::PathBuf,
}
```

说明：

1. `ScreenshotMode` 用于统一业务入口
2. `ScreenshotRequest` 负责描述目标窗口、区域和输出文件名
3. `CaptureRegion` 统一裁剪矩形定义
4. `ScreenshotResult` 用于 UI 反馈和后续模块复用

## 5.1 输出目录约定

截图输出不弹保存对话框，统一直接写入项目根目录下的 `screenshots/` 文件夹。

规则如下：

1. 若 `screenshots/` 不存在，则运行时自动创建
2. 所有截图结果统一保存为 PNG
3. 文件名由截图模块自动生成，避免覆盖已有文件

建议文件名格式：

1. `window_YYYYMMDD_HHMMSS.png`
2. `region_render_YYYYMMDD_HHMMSS.png`
3. `desktop_YYYYMMDD_HHMMSS.png`
4. `screen_region_YYYYMMDD_HHMMSS.png`

这样可以保证：

1. 不和报告输出目录混在一起
2. 联调和测试时可以直接在固定目录查看结果
3. 后续若要接自动清理策略，也容易集中处理

## 6. 程序内截图实现方案

### 6.1 程序窗口截图

建议基于 Bevy 当前窗口的渲染结果做 GPU 纹理读回，再编码为 PNG。

流程如下：

1. 用户点击“截图”
2. UI 发送 `ScreenshotRequest::Window`
3. 截图模块定位目标窗口对应的渲染目标
4. 渲染完成后将纹理读回 CPU 内存
5. 将像素缓冲区编码为 PNG
6. 保存到项目根目录下的 `screenshots/`
7. 返回 `ScreenshotResult`

优点：

1. 不依赖系统 API
2. 画面内容和程序实际渲染保持一致
3. 适合后续和报告模块统一对接

风险点：

1. GPU 读回通常存在一帧延迟
2. 不能设计成按钮点击后立即同步保存
3. 多窗口时要明确主窗口、弹窗窗口的目标选择逻辑

### 6.2 程序内区域截图

程序内固定区域截图建议支持两条实现路径：

1. 独立 render 导出
2. 整窗截图后裁剪

其中，固定功能区截图的优先方案建议是独立 render，整窗裁剪作为通用兜底方案保留。

#### 6.2.1 优先方案：独立 render 导出

适用前提：

1. 该功能区本身有独立的相机、纹理或离屏渲染目标
2. 该功能区的显示内容可以和主窗口 UI 解耦
3. 希望导出的内容更干净，不受外层遮挡、高亮、临时覆盖层影响

典型适用模块：

1. 医学影像主视图
2. 波形绘图区
3. 图表区域
4. 后续可独立维护 render target 的功能区

建议流程如下：

1. 业务模块为固定功能区注册截图描述信息
2. 若该区域具备独立 render target，则直接导出对应纹理
3. 渲染完成后将纹理读回 CPU 内存
4. 编码并保存为 PNG 到 `screenshots/`

优点：

1. 导出内容更聚焦，不会把外层 UI 杂项一起截进去
2. 更适合报告、图像分析、结果归档
3. 对固定功能区的语义更稳定

边界：

1. 不是所有区域都天然具备独立 render target
2. 仍然存在 GPU 读回带来的异步完成问题
3. 如果独立 render 的更新节奏和主界面不同，可能与用户当前看到的结果存在一帧差异

#### 6.2.2 兜底方案：整窗截图后裁剪

对于没有独立 render target 的固定区域，建议回退到“程序窗口截图结果 + CPU 侧裁剪”。

流程如下：

1. 先获取完整窗口截图原图
2. 根据程序内固定区域坐标裁剪像素数据
3. 将裁剪结果编码并保存为 PNG 到 `screenshots/`

优点：

1. 通用性强
2. 截图结果与用户当前看到的最终合成画面更一致
3. 不要求业务模块必须提前准备独立 render target

不足：

1. 可能把额外 UI 装饰、遮挡、高亮态一起截进去
2. 对结果纯净度不如独立 render

#### 6.2.3 设计建议

首版建议直接收敛为固定区域导出，并按以下原则处理：

1. 能独立 render 的区域，优先走独立 render 导出
2. 不能独立 render 的区域，走整窗截图后裁剪
3. 统一对外暴露同一套固定区域截图接口，业务层不感知底层走哪条路径

建议首批固定区域包括：

1. 医学影像主视图区域
2. 波形绘图区区域
3. 图表区域
4. 其他由业务模块注册的固定区域

## 7. 系统级截图实现方案

### 7.1 程序所在桌面截图

建议封装独立的 Windows 抓屏后端：

- 优先复用成熟 crate
- 若现成 crate 稳定性不足，再直接封装 Win32 API

流程如下：

1. 用户点击“程序所在桌面截图”
2. 先定位当前程序主窗口所在的显示器
3. 调用系统级抓屏接口获取该显示器的桌面图像
3. 将原始像素缓冲区转换为 PNG
4. 保存到 `screenshots/`

注意事项：

1. 需要从程序窗口句柄或窗口坐标反查所属显示器
2. 不处理跨显示器窗口截图拼接
3. 需要处理当前显示器的物理像素坐标
4. 需要统一输出为 PNG，避免不同模式结果格式不一致

### 7.2 鼠标选定区域截图

建议采用“整屏抓图 + 顶层遮罩框选 + 裁剪保存”的实现方式。

这里的“整屏抓图”也指当前程序所在显示器的整屏图像，而不是全部显示器拼接。

流程如下：

1. 先抓取一帧完整桌面图像
2. 弹出无边框半透明顶层窗口作为遮罩层
3. 用户拖拽选择矩形区域
4. 根据选区从整屏图像中裁剪目标区域
5. 保存为 PNG 到 `screenshots/`

这样做的原因是：

1. 实现路径清晰
2. 裁剪逻辑可与程序内区域截图复用
3. 交互层与抓图层可以明确分离

## 8. UI 交互建议

建议单独增加一个截图测试页面，用于联调和验收截图能力。

页面建议位置：

- `ui/src/homepage/screenshot/`

测试页面建议包含以下入口：

1. 程序窗口截图
2. 程序内区域截图
3. 程序所在桌面截图
4. 桌面框选截图

同时页面内建议固定放置两个测试功能区：

1. 测试区 A：独立 render 区
2. 测试区 B：整窗裁剪区

测试区 A 建议特征：

1. 具备独立相机或独立 render target
2. 内容包含动态图形、文字、水印或边框
3. 专门用于验证独立 render 导出路径

测试区 B 建议特征：

1. 使用普通 UI 区域实现
2. 不提供独立 render target
3. 内容包含文本、按钮、背景块、边框等元素
4. 专门用于验证整窗截图后裁剪路径

建议交互规则：

1. 截图前先选择模式
2. 截图后直接保存到项目根目录 `screenshots/`
3. 保存成功后在界面内提示文件路径
4. 保存失败时展示明确错误信息

程序内区域截图建议增加：

1. 固定区域名称列表
2. 当前区域预览高亮
3. 截图结果尺寸提示
4. 区域不可用时给出明确禁用原因

## 9. 与现有模块的关系

截图模块建议作为统一图片输出基础设施，供以下模块复用：

1. `medical_image`
2. `signal_processing`
3. `report_generator`
4. 后续问题反馈与日志附件功能

推荐约束：

1. 截图模块统一产出 PNG
2. 业务模块不直接关心底层抓图细节
3. 报告模块只读取现成 PNG，不参与截图流程

这样可以让截图功能保持边界清晰，也能减少不同模块重复实现导图逻辑。

## 10. 实现步骤

### Phase 1: 跑通截图测试页与程序窗口截图闭环

1. 新增 `screenshot` crate，并加入 workspace
2. 创建 `screenshot/src/lib.rs`、`error.rs`、`request.rs`、`result.rs`、`save.rs`
3. 在 `save.rs` 中封装统一输出目录逻辑，固定写入项目根目录 `screenshots/`
4. 实现 `screenshots/` 目录不存在时自动创建
5. 实现统一 PNG 文件名生成规则，避免覆盖已有文件
6. 新增 `ui/src/homepage/screenshot/` 模块，并创建 `components.rs`、`constants.rs`、`resources.rs`、`systems.rs`、`plugin.rs`
7. 在首页功能页中增加截图测试页面入口
8. 在截图测试页面中放置 4 个按钮，分别对应 4 种截图方式
9. 优先打通“整个程序截图”按钮
10. 在 `screenshot` crate 中实现程序窗口截图的最小闭环
11. 完成窗口截图结果读回、PNG 编码与写盘
12. 在测试页面展示截图成功/失败状态和输出路径

### Phase 2: 固定区域截图与双路径验证

1. 在截图测试页面中增加两个固定测试区
2. 实现 `测试区 A`，用于验证独立 render 导出
3. 为 `测试区 A` 配置独立相机、纹理或离屏 render target
4. 在 `screenshot` crate 中增加固定区域注册模型
5. 定义固定区域元信息，包括区域 ID、区域名称、截图方式、坐标或 render target 信息
6. 实现独立 render 导出路径
7. 将独立 render 纹理读回 CPU 并保存到 `screenshots/`
8. 实现 `测试区 B`，用于验证整窗截图后裁剪
9. 为 `测试区 B` 配置固定区域矩形定义，但不提供独立 render target
10. 实现整窗截图后裁剪的兜底路径
11. 在测试页面增加固定区域选择或按钮映射，明确区分 A 区和 B 区截图
12. 验证两条路径都能正确输出 PNG，且输出文件名可区分来源

### Phase 3: 程序所在桌面截图与桌面框选截图

1. 在 `screenshot/src/win_capture.rs` 中封装 Windows 桌面抓图能力
2. 实现根据程序主窗口定位当前所属显示器
3. 打通“程序所在桌面截图”按钮
4. 实现当前显示器桌面图像抓取并保存到 `screenshots/`
5. 设计桌面框选截图的遮罩层交互流程
6. 实现桌面整屏抓取，范围限定为程序所在显示器
7. 实现无边框半透明遮罩窗口
8. 实现鼠标按下、拖拽、释放后的矩形选区逻辑
9. 实现根据选区裁剪桌面图像并保存为 PNG
10. 处理当前显示器下的 DPI 坐标换算
11. 明确不支持跨显示器窗口和跨显示器框选
12. 完成 4 种截图方式在测试页面中的联调与验收

## 11. 风险与边界

主要风险如下：

1. Bevy 渲染结果读回不是同步即时完成，流程需要异步化
2. 高 DPI 下程序逻辑坐标和物理像素坐标可能不一致
3. 多窗口场景下要明确截图目标窗口
4. 系统级截图与程序内截图不能混用同一套坐标假设
5. 多显示器环境下要正确判断程序当前所属显示器
6. 独立弹窗、系统对话框、原生菜单可能无法被程序内截图覆盖

本方案当前不包含：

1. 复制到剪贴板
2. 截图后标注绘制
3. OCR 文本识别
4. 跨平台统一实现

这些能力建议作为后续扩展，不纳入首版范围。

## 12. 需求映射

对应 `docs/requirements.md` 中的“截图”需求，建议细化为：

1. 程序窗口截图
2. 程序内区域截图
3. 程序所在桌面截图
4. 桌面框选截图
5. 保存为 PNG

相较于原始表述，这样可以避免“区域截图”到底是程序内还是桌面的歧义，也更方便后续拆分任务和验收。

其中：

1. `区域截图` 指程序内固定区域截图，不包含程序内拖拽框选
2. 用户拖拽选区只保留在桌面级截图场景中
