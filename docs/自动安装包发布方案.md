# 自动安装包发布方案

## 1. 背景

当前仓库已经具备：

1. `Windows` 发布工作流
2. `entry.exe` 的 release 构建
3. `installer/scripts/build_installer.ps1` 安装包构建脚本
4. `WiX v5` 生成 `msi` 的基础能力

但当前 `Release` 工作流还停留在：

1. tag 推送后校验格式、clippy、测试
2. 构建 `entry.exe`
3. 上传 `entry.exe`
4. 创建 GitHub Release

也就是说，版本提升后还不能自动产出安装包，安装包构建仍依赖本地手工执行。

这会带来几个实际问题：

1. 发布流程不完整，`GitHub Release` 里只有裸 `exe`
2. 安装包是否可构建、是否缺少运行时、模型是否齐全，无法在发布链路里自动验证
3. 每次发版都需要手工介入，容易漏步骤
4. 本地环境和 CI 环境差异会导致“本地能打包，远端不能打包”

## 2. 目标

希望在“提升版本并打 tag”后，自动完成：

1. 运行发布前校验
2. 构建 `entry` release 程序
3. 整理 `GStreamer` 运行时
4. 整理 `CUDA` 运行时
5. 整理 `Whisper` 模型目录
6. 生成 `msi` 安装包
7. 把 `msi` 和必要构建产物上传到 `GitHub Release`

同时要求：

1. 尽量复用现有 `installer/scripts/build_installer.ps1`
2. 不在 workflow 里重复手写一套安装包逻辑
3. 失败时能快速定位是环境问题、模型问题、WiX 问题还是脚本问题

## 3. 非目标

这次方案先不做：

1. 自动签名
2. 自动在线更新
3. 多平台安装包
4. 在 CI 中动态下载大型模型权重
5. 把所有安装资源都存进仓库

## 4. 当前阻塞点

要把安装包接进 CI/CD，当前至少有这几个阻塞：

1. `build_installer.ps1` 依赖本机 `installer/local_paths.ps1`
2. `prepare_models.ps1` 依赖工作区已有 `whisper_base` 和 `whisper-large-v3`
3. `prepare_cuda_runtime.ps1` 依赖本机 `CUDA Toolkit` 路径
4. `build_installer.ps1` 依赖本机已安装 `WiX` 命令和扩展
5. `Release` workflow 目前只上传 `entry.exe`，没有 staging 和 `msi` 产物链路

这些都属于“本地可用，但 CI 环境尚未标准化”的典型问题。

## 5. 设计结论

最终选择：

1. 采用方案 2
2. 不在 CI 中临时从 Hugging Face、CUDA 官方来源现下大资源
3. 改为从自有托管的大资源包地址下载固定版本资源
4. 资源来源优先使用 `GitHub Release assets`

建议把自动安装包发布接入现有 `.github/workflows/release.yml`，而不是另起一套平行发布流程。

发布触发条件仍保持不变：

1. 推送形如 `v1.2.3` 的 tag

但 workflow 需要从“构建 exe 并发 Release”升级为“校验 -> 构建程序 -> 准备安装资源 -> 生成 MSI -> 发 Release”。

核心原则：

1. `installer/scripts/build_installer.ps1` 作为唯一安装包入口
2. workflow 只负责准备 CI 所需环境变量和第三方运行时路径
3. 所有安装资源统一进入 staging，再由 `WiX` 打包
4. Release 资产至少包含 `msi`
5. 大资源包由单独的资源版本维护，不和每次代码 tag 强绑定下载来源

## 6. 方案 2 说明

方案 2 的核心思路是：

1. 先把安装包需要的大资源整理成固定压缩包
2. 上传到你自己控制的地址
3. 正式 release workflow 只下载这些压缩包
4. 校验通过后解压并调用现有安装包脚本

这里的“大资源”包括：

1. `Whisper Base` 模型必需文件
2. `Whisper Large V3` 模型必需文件
3. `CUDA` 可再分发运行时 `dll`
4. `GStreamer` 运行时

这样做的原因很直接：

1. 模型和运行时体积大，CI 临时下载慢
2. 外部来源波动大，容易限流、超时或失败
3. 发版流程应该稳定，不能被大文件下载拖垮

## 7. 推荐资源包拆分

建议至少拆成 4 类资源：

1. `whisper-base-assets-<version>.zip`
2. `whisper-large-v3-files-<version>.zip` 与 `whisper-large-v3-model.partNN`
3. `cuda-runtime-<version>.zip`
4. `gstreamer-runtime-*.zip`

这样拆分不是为了形式统一，而是为了绕开 `GitHub Release assets` 的单文件大小限制。

其中 `whisper-base-assets-<version>.zip` 内部建议结构：

```text
whisper_base/
├── config.json
├── model.safetensors
├── tokenizer.json
└── ...
```

`whisper-large-v3-files-<version>.zip` 只放小文件：

```text
whisper/
└── whisper-large-v3/
    ├── config.json
    ├── tokenizer.json
    ├── vocab.json
    └── ...
```

`whisper-large-v3-model.partNN` 用来承载拆分后的 `model.safetensors`。

`cuda-runtime-<version>.zip` 内部建议结构：

```text
cuda-runtime/
└── bin/
    ├── cudart*.dll
    ├── cublas*.dll
    ├── cublasLt*.dll
    └── ...
```

`gstreamer-runtime-*.zip` 建议按目录拆分：

```text
gstreamer-runtime-bin-<version>.zip
gstreamer-runtime-plugins-<version>.zip
gstreamer-runtime-libexec-<version>.zip
```

## 8. 资源托管位置建议

优先推荐：

1. 单独的 `GitHub Release assets`

也就是专门建一个资源版本 Release，例如：

1. tag: `installer-assets-2026-03-27`
2. assets:
   1. `whisper-base-assets-2026-03-27.zip`
   2. `whisper-large-v3-files-2026-03-27.zip`
   3. `whisper-large-v3-model.part01`
   4. `whisper-large-v3-model.part02`
   5. `cuda-runtime-13.0.zip`
   6. `gstreamer-runtime-bin-1.26.9.zip`
   7. `gstreamer-runtime-plugins-1.26.9.zip`
   8. `gstreamer-runtime-libexec-1.26.9.zip`

这样做的好处：

1. 不进入 git 仓库历史
2. URL 固定
3. 版本可追踪
4. 便于 CI 直接下载

如果后续需要更强控制，也可以迁移到：

1. 私有对象存储
2. 公司文件服务器
3. 单独的资源仓库 Release

## 9. 资源清单文件设计

建议在仓库中增加一份资源清单文件，例如：

1. `installer/manifests/release_assets.json`

内容建议至少包含：

1. 资源名称
2. 资源版本
3. 下载 URL
4. `SHA256`

## 10. 推荐流程

建议把 `release.yml` 调整为 4 个阶段：

### 6.1 `verify`

职责：

1. `cargo fmt --check`
2. `cargo clippy`
3. `cargo test`
4. 额外检查安装包脚本是否可调用

说明：

1. 继续复用当前 `verify` job
2. 保证发 tag 的提交本身是可发布状态

### 10.2 `prepare-assets`

职责：

1. 读取资源清单
2. 下载 `whisper-base zip`
3. 下载 `whisper-large-v3` 小文件 zip 和模型分片
4. 下载 `cuda-runtime zip`
5. 下载 `gstreamer` 多个 zip
6. 校验每个文件的 `SHA256`
7. 解压到约定目录
8. 把 `whisper-large-v3` 分片重新组装成 `model.safetensors`
9. 生成 `installer/local_paths.ps1` 或直接通过参数把路径传给脚本

设计建议：

1. CI 不直接访问 Hugging Face 拉大模型
2. CI 不直接从 `CUDA Toolkit` 官网临时拼装运行时
3. 所有来源统一落到你自己维护的固定资源版本
4. 下载失败时能明确定位到是哪一个资源包失败

### 10.3 `build-installer`

职责：

1. 安装 Rust toolchain
2. 安装 `WiX`
3. 安装 `WixToolset.UI.wixext`
4. 调用 `installer/scripts/build_installer.ps1`
5. 产出 `installer/dist/*.msi`

关键点：

1. 直接复用现有 `build_installer.ps1`
2. 通过参数显式传入：
   1. `-WorkspaceRoot`
   2. `-GStreamerRoot`
   3. `-CudaBinRoot`
   4. `-StageRoot`
   5. `-OutputRoot`
3. 不依赖 CI 机器上的手工配置文件

### 10.4 `create-release`

职责：

1. 下载 `msi` 构建产物
2. 从 tag 提取版本号
3. 创建或更新 GitHub Release
4. 上传 `msi`
5. 可选上传 `entry.exe`、staging 清单、构建日志

Release 最少应包含：

1. `rust_packages_survey-x.y.z.msi`

建议额外包含：

1. `entry.exe`
2. `SHA256SUMS.txt`

## 11. CI 环境准备方案

### 7.1 `WiX`

CI 中显式安装：

1. `dotnet tool install --global wix --version 5.0.2`
2. `wix extension add --global WixToolset.UI.wixext/5.0.2`

这样就不依赖 runner 的预装状态。

### 11.2 `GStreamer`

CI 不再依赖 `Chocolatey` 临时安装 `gstreamer` 作为安装资源来源。

改为：

1. 下载预制 `gstreamer` 多个 zip
2. 解压后把目录传给 `build_installer.ps1`

### 11.3 `CUDA`

CI 不再要求 runner 预装完整 `CUDA Toolkit`，也不在正式 release 里临时下载 `CUDA` 大文件。

改为：

1. 下载预制 `cuda-runtime zip`
2. 解压后把 `bin` 目录传给 `build_installer.ps1`
3. 只使用事先确认可再分发的 `dll` 集合

### 11.4 `Whisper` 模型

不能要求 CI runner 自带模型，也不建议在正式发布 workflow 中临时从 Hugging Face 现拉。

改为：

1. 下载预制 `whisper-base zip`
2. 下载 `whisper-large-v3` 小文件 zip
3. 下载并组装 `model.safetensors` 分片
4. 直接传给安装包脚本

## 12. 对现有脚本的改造建议

为了让 CI 更稳定，建议后续补这几个改造：

### 8.1 `build_installer.ps1`

需要做到：

1. 所有外部依赖都可通过参数传入
2. 不依赖 `installer/local_paths.ps1`
3. 缺少参数时给出明确错误
4. 输出最终 `msi` 路径

### 12.2 `prepare_models.ps1`

需要做到：

1. 支持显式传入 `Whisper Base` 模型目录
2. 支持显式传入 `Whisper Large V3` 模型目录
3. 不依赖默认工作区目录一定存在模型

### 12.3 新增 `fetch_release_assets.ps1`

职责：

1. 读取 `release_assets.json`
2. 下载多个资源 zip 和大文件分片
3. 校验 `SHA256`
4. 解压到工作目录
5. 组装分片文件
6. 输出解压后的目录路径

### 12.4 新增 `prepare_release_assets.ps1`

职责：

1. 统一调用资源包下载、校验、解压和安装包构建
2. 作为 release workflow 的唯一脚本入口

如果这样做，workflow 会更薄，排障也更集中。

## 13. 合规与分发边界

### 13.1 `CUDA`

不能随意把整个 `CUDA Toolkit` 打包上传。

方案 2 下必须坚持：

1. 资源包里只放明确可再分发的 `CUDA` 运行时 `dll`
2. 不包含 `NVIDIA` 驱动
3. 附带对应许可证文本

### 13.2 `GStreamer`

不能默认把所有插件都打进资源包。

方案 2 下必须坚持：

1. 只打包项目实际需要的运行时和插件
2. 对插件集合做许可证白名单
3. 对可能涉及编解码专利的部分单独审查

### 13.3 模型

模型资源包要固定 revision，避免同一个资源版本对应不同文件内容。

要求：

1. 资源包版本固定
2. 每个 zip 都有 `SHA256`
3. 资源清单变更要和代码一起进入评审

## 14. workflow 改造建议

建议把 `.github/workflows/release.yml` 从当前三段式：

1. `verify`
2. `build-windows`
3. `create-release`

升级为：

1. `verify`
2. `prepare-assets`
3. `build-installer`
4. `create-release`

其中：

1. `build-windows` 可以并入 `build-installer`
2. `entry.exe` 不再是唯一发布产物
3. `msi` 成为主发布产物

## 15. 版本号策略

建议安装包文件名包含 tag 版本号，例如：

1. `rust_packages_survey-0.1.1.msi`

Release 名称继续沿用：

1. `Release v0.1.1`

这样用户下载时不需要猜版本，发布资产也更清晰。

## 16. 风险与应对

### 16.1 资源包过大

风险：

1. 首次上传和下载资源包仍然耗时
2. `GitHub Release assets` 下载速度受地区和网络影响

应对：

1. 尽量按功能拆包
2. 不在每次构建中重复生成资源包
3. 必要时迁移到更稳定的对象存储

### 16.2 资源版本漂移

风险：

1. 代码版本和资源包版本不匹配
2. 同一资源 tag 被重复覆盖，导致内容漂移

应对：

1. 资源版本只增不改
2. 资源 zip 必须校验 `SHA256`
3. 代码仓库中固定记录资源版本号

### 16.3 Release assets 被误删或权限变化

风险：

1. CI 下载地址失效
2. 私有资源权限调整后 workflow 无法访问

应对：

1. 资源仓库权限单独管理
2. 正式发版前增加资源可达性检查
3. 必要时保留镜像地址

### 16.4 WiX 环境差异

风险：

1. 本地能打包，CI 中因为工具版本不同失败

应对：

1. workflow 中显式安装固定版本 `WiX`
2. 不依赖 runner 预装状态

### 16.5 Release 工件过大

风险：

1. 安装包、模型、日志等产物会拉高 Actions artifact 体积

应对：

1. Release 只上传 `msi` 和必要校验文件
2. 中间 staging 产物只做短期 artifact 保留

## 17. 资源清单

如果你决定采用方案 2，建议实际维护这一份资源清单：

1. `whisper-base-assets-<date>.zip`
   1. `whisper_base/`
2. `whisper-large-v3-files-<date>.zip`
   1. `whisper/whisper-large-v3/` 中除 `model.safetensors` 外的文件
3. `whisper-large-v3-model.partNN`
   1. 按顺序拆分后的大模型文件
4. `cuda-runtime-<version>.zip`
   1. 当前程序实际依赖的可再分发 `dll`
5. `gstreamer-runtime-*.zip`
   1. 已确认许可证和插件范围的运行时目录
6. `release_assets.json`
   1. 所有资源文件的 URL
   2. 所有资源文件的 `SHA256`
   3. 资源版本与组装规则

## 18. 实施顺序

建议按以下顺序推进：

1. 先整理三份预制资源包
2. 把资源包上传到 `GitHub Release assets`
3. 新增 `release_assets.json`
4. 把 `build_installer.ps1` 改成完全参数化
5. 新增 `fetch_release_assets.ps1`
6. 在本地模拟“只下载资源包并打安装包”
7. 再改 `.github/workflows/release.yml`
8. 最后用测试 tag 验证 GitHub Release 是否自动带出 `msi`

## 19. 最终结论

建议把“自动安装包发布”并入当前 tag 触发的 `Release` 工作流，并坚持两条原则：

1. 安装包生成逻辑只保留一套，以 `installer/scripts/build_installer.ps1` 为准
2. CI 不临时现下大模型和大运行时，而是只消费预制资源包

落地后，版本提升流程会从现在的：

1. 打 tag
2. 自动上传 `entry.exe`

升级为：

1. 打 tag
2. 自动校验
3. 自动下载并校验预制资源包
4. 自动构建 `msi`
5. 自动发布 GitHub Release 安装包

这样发布链路才算真正闭环。
